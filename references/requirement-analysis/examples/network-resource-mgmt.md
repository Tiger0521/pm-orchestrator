### 问题本质

运维工程师为新项目申请服务器、IP、域名时，需要在多个系统间反复填写信息、等待审批，平均耗时 3-5 天，且容易因信息不一致被打回。

### 目标用户画像

- **角色**：运维工程师、项目经理
- **场景**：新项目上线前需要申请网络资源
- **痛点**：信息重复填写、审批状态不透明、打回后不知道改哪里

### 当前替代方案及痛点

| 替代方案 | 优点 | 缺点 |
|----------|------|------|
| 邮件 + Excel | 灵活 | 信息版本混乱，审批链路长 |
| 工单系统 | 有记录 | 仍需跨系统查资源状态 |

### 关键假设

1. 审批规则可以结构化配置
2. 用户愿意在统一入口提交申请
3. 资源状态可以通过 API 同步

### 成功指标

1. 资源申请平均耗时从 4 天降到 1 天
2. 信息打回率从 30% 降到 5% 以下
3. 用户满意度 ≥ 4.0/5

### 产品定位

为运维和项目经理提供一站式的网络资源申请、审批、查询平台，让资源申请从 4 天缩短到 1 天。

### 战略价值

- 减少跨系统协作成本
- 提升资源申请透明度
- 降低人工审批错误率

### 目标用户 / 角色

| 角色 | 描述 | 核心诉求 |
|------|------|---------|
| 申请人 | 运维/项目经理 | 快速提交、实时看进度 |
| 审批人 | 网络管理员/主管 | 规则清晰、一键审批 |
| 资源管理员 | 资产管理专员 | 资源状态可追踪 |

### 核心场景

#### 场景一：提交资源申请

申请人填写申请表单，系统自动校验规则并提交到对应审批人。

#### 场景二：审批资源申请

审批人收到通知，查看申请详情，一键通过或打回并注明原因。

### 边界

**范围内**：
- 服务器、IP、域名三类资源申请
- 申请、审批、查询三个核心环节

**范围外**：
- 物理资源采购流程
- 财务预算审批
- 机房现场操作

### 成功指标

1. 申请平均耗时 ≤ 1 天
2. 打回率 ≤ 5%
3. 上线 3 个月内覆盖 80% 新项目

### Feature 描述

提供统一的资源申请表单，根据资源类型动态展示字段，自动校验必填项和规则冲突。

### 用户价值

减少信息重复填写，降低因信息错误导致的审批打回。

### 业务流程

```
申请人进入 → 选择资源类型 → 填写表单 → 系统校验 → 提交申请 → 等待审批
```

### 业务规则

| 规则项 | 规则说明 |
|--------|---------|
| 必填校验 | 资源类型、项目名称、用途说明为必填 |
| 冲突校验 | 同一 IP 不能重复申请 |
| 自动填充 | 已登录用户自动填充申请人信息 |

### 优先级

P0

### 依赖

- 用户认证系统
- 资源基础数据（IP 池、服务器清单）

### 验收标准

1. 表单支持服务器、IP、域名三类资源
2. 必填项未填写时阻止提交并提示
3. 提交后申请人可在列表中查看状态
