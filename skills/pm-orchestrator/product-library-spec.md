# 产品库规范与匹配算法

> 本文件定义全局产品库的目录结构规范和产品匹配算法。需求分析 intake 在创建项目记录、且尚未确定 projectType 时引用本文件；候选导览、复用引导和用户追问规则以 `references/requirement-analysis/instruction.md` 为准。

---

## 1. 产品库目录结构

全局路径：`~/.product-library/`（Windows: `C:\Users\<用户名>\.product-library\`）。

用户须将此目录初始化为 git 仓库并上传远程仓库，支持多人协作更新。

```
~/.product-library/
├── _manifest.md                         # 全局清单
└── <product-id>/                        # 产品文件夹，kebab-case
    ├── _product.md                      # 产品元信息
    ├── requirement-analysis/
    │   ├── requirement-cards/
    │   ├── epics/
    │   └── features/
    └── requirement-breakdown/
        ├── user-stories/
        └── traceability-matrices/
```

### 命名规则

| 层级 | 规则 | 示例 |
|------|------|------|
| 产品文件夹 | `^[a-z0-9][a-z0-9-]{0,62}$` | `network-resource-mgmt` |
| 元信息文件 | `_` 前缀 + `.md` | `_manifest.md`、`_product.md` |
| 阶段文件夹 | 与 skill 阶段名对齐 | `requirement-analysis/`、`requirement-breakdown/` |
| 文档类型文件夹 | 复数 | `requirement-cards/`、`epics/`、`features/` |
| 文档文件 | `<前缀>-<序号>.md` | `req-001.md`、`epic-001.md` |

### `_manifest.md` 格式

```yaml
---
version: "1.0"
lastUpdated: "2026-07-15T10:00:00Z"
products:
  - id: "network-resource-mgmt"
    name: "网络资源全生命周期管理系统"
    summary: "面向网络设备管理部门的全生命周期管理系统"
    keywords: ["网络资源", "全生命周期", "数据治理"]
    documentCount:
      requirementCards: 1
      epics: 1
      features: 3
      userStories: 8
      traceabilityMatrices: 1
    lastUpdated: "2026-07-10T14:30:00Z"
---
```

### `_product.md` 格式

```yaml
---
id: "network-resource-mgmt"
name: "网络资源全生命周期管理系统"
summary: "面向网络设备管理部门的全生命周期管理系统"
keywords: ["网络资源", "全生命周期", "数据治理"]
businessDomain: "网络资源管理"
createdAt: "2026-07-01T10:00:00Z"
lastUpdated: "2026-07-10T14:30:00Z"
---
```

产品库中文档格式与 skill 正式产物完全相同（YAML frontmatter + 按写作范式撰写的正文），不设 `status` 字段。

## 2. 只读规则

- skill 的任何阶段都不得写入产品库
- 项目正式文档只写入项目目录 `docs/`，不写入产品库
- 产品库文档视为**已确认产品资产**，subagent 将其中的产品定义视为硬约束
- 产品库通过 `export-to-library.sh` 脚本 + git 手动维护

---

## 3. 项目类型定义

| projectType | 产品库文档读取 | 文档策略 |
|-------------|--------------|---------|
| `new` | 不读取 | 产出新的需求卡片、Epic、Feature |
| `iteration` | 读取 Epic + Feature | 可新增 Feature，引用已有 Epic |
| `refactor` | 读取 Epic + Feature + User Story | Epic、Feature、User Story 均不变，聚焦非功能性需求 |

---

## 4. 匹配维度与权重

| 维度 | 权重 | 用户侧来源 | 产品库侧来源 |
|------|------|-----------|------------|
| D1 问题本质 | 30% | 用户描述中的问题陈述 | 需求卡片"问题本质还原"字段 |
| D2 痛点 | 25% | 用户描述中的痛点陈述 | 需求卡片"痛点"字段 |
| D3 现状描述 | 15% | 用户描述中的现状陈述 | 需求卡片"现状描述"字段 |
| D4 业务场景 | 15% | 用户描述中的场景陈述 | Epic"核心场景"字段 |
| D5 用户角色 | 10% | 用户描述中的角色陈述 | Epic"用户角色"字段 |
| D6 业务领域 | 5% | 用户描述的整体领域 | `_product.md` 的 `businessDomain` 和 `keywords` |

权重设计理由：D1 决定产品方向，权重最高；D2 次高，痛点相似说明问题表现一致；D3/D4 中等，说明业务上下文重叠；D5 低，角色可扩展；D6 最低，仅作初筛信号。

---

## 5. 语义等价分级

所有维度的匹配是**语义比对**，不是字面匹配。

| 等级 | 分数 | 定义 | 示例 |
|------|------|------|------|
| 等价 | 100 | 同一概念不同表述 | "大学老师" ≈ "研究员" |
| 包含 | 75 | 子集或超集关系 | "设备管理" ⊂ "设备全生命周期" |
| 相关 | 50 | 同一领域但不完全相同 | "网络设备" ⊂ "IT 设备" |
| 弱相关 | 25 | 有间接关联但不直接对应 | "资产管理" ~ "网络资源管理" |
| 无关 | 0 | 无任何语义关联 | "餐饮管理" vs "网络资源管理" |

### 语义判断角度

- **角色（D5）**：是否同职业类别、是否上下级关系、是否职能重叠
- **场景（D4）**：是否同业务环节、是否上下游关系、是否同流程不同步骤
- **问题本质（D1）**：是否同类业务断点（信息不对称/流程断点/数据缺失/决策滞后/协作障碍/资源浪费）、是否指向同根因
- **痛点（D2）**：是否同类痛点（返工/等待/错误/成本/风险）、是否同根因导致、影响对象和频次是否相似
- **现状（D3）**：当前流程是否相似、是否用类似替代方案（Excel/群聊/邮件）、瓶颈环节是否相同
- **领域（D6）**：是否同行业或业务域、是否上下游关系、keywords 是否语义重叠

## 6. 评分与匹配度

### 总分计算

```
总分 = D1×0.30 + D2×0.25 + D3×0.15 + D4×0.15 + D5×0.10 + D6×0.05
```

### 匹配度分级

| 匹配度 | 总分范围 | 含义 |
|--------|---------|------|
| high | ≥75 | 高度重叠，优先向用户解释已有产品能力并核对本次需求事实 |
| medium | 50-74 | 部分重叠，建议查看已有产品 |
| low | 25-49 | 弱关联，倾向新建但保留候选关联 |
| none | <25 | 无关联，倾向新建并等待用户确认 |

---

## 7. 部分匹配处理规则

按维度重要性分层判断，不简单取平均。

| 情形 | 条件 | 结论 |
|------|------|------|
| 核心匹配 | D1 ≥ 75 且 D2 ≥ 75 | ≥ medium（其他差异是扩展空间） |
| 核心匹配+角色扩展 | D1 ≥ 75 且 D2 ≥ 75 且 D5 < 50 | high（若有 3+ 维度 ≥ 75）或 medium |
| 问题对但痛点不同 | D1 ≥ 75 且 D2 < 50 | medium（同一问题不同表现） |
| 痛点对但问题不同 | D1 < 50 且 D2 ≥ 75 | low（症状相似根因不同） |
| 多数匹配个别偏离 | 5+ 维度 ≥ 75 且 1 维度 = 0 | 仍为 high（偏离维度是扩展点） |
| 中等重叠 | 3-4 维度 ≥ 75 | medium |
| 少量重叠 | 1-2 维度 ≥ 75 | low |

### 特殊处理

- **角色不匹配（D5 低但其他高）**：不降低匹配度，标注"可作为产品角色扩展"，下一轮核对目标用户、场景和新增能力
- **场景部分重叠（D4 = 50）**：不降低匹配度，列出重叠和不重叠场景，让用户确认范围
- **领域不匹配但问题本质匹配（D6 低但 D1 高）**：不低于 medium，标注"可复用设计思路"，下一轮核对业务边界和产品复用价值
- **全部中等（全 50-74）**：medium，说明"各方面部分重叠但不完全匹配"，让用户决定

## 8. 匹配流程

```
Step 0：前置校验
  validate-product-library.sh 校验产品库结构 → 通过才继续
         ▼
Step 1：读取全局清单
  读取 ~/.product-library/_manifest.md，获取所有产品元信息
  产品数为 0 → 匹配结果 none，建议项目类型为 new 并等待用户确认
         ▼
Step 2：领域初筛（D6）
  用户已确认的初始描述 + 前置背景摘要 vs businessDomain + keywords 做语义比对
  D6 = 0 的产品直接排除；若用户背景材料显示跨域复用可能，可保留为 low 并标注理由
         ▼
Step 3：深度匹配（D1-D5）
  对每个通过初筛的产品：
  a. 读取 requirement-cards/ → 评分 D1（问题本质）、D2（痛点）、D3（现状）
  b. 读取 epics/ → 评分 D4（场景）、D5（角色）
         ▼
Step 4：计算总分与分级
  按权重计算总分，按 §7 修正分级
  每个维度记录分数和判断理由
         ▼
Step 5：排序与筛选
  按总分降序，过滤 none，保留 low 及以上
         ▼
Step 6：形成候选产品导览
  按匹配度从高到低呈现候选关联产品；若候选超过 1 个，先展示全方位导览，让用户选择先了解哪个
  导览内容包括：解决的问题、可能重叠的业务闭环、可能扩展的方向、建议优先级
         ▼
Step 7：进入需求分析 intake 的复用引导
  按 requirement-analysis instruction 的“产品匹配与复用引导”执行
  递进顺序固定为：需求卡片 → Epic → Feature
  每层批量展示多个字段状态，只问一个最高价值问题；问题围绕用户自己的业务目标、场景、角色、数据、规则、流程、验收口径展开
  用户补充或纠正后，由系统更新对应维度评分、覆盖点、差异点和扩展假设
         ▼
Step 8：形成项目类型建议
  汇总“已有产品已覆盖 / 本次新增或变化 / 仍待确认”，再给出项目类型建议：
  iteration → 差异主要可解释为已有产品的角色/对象/规则/数据源/流程/场景/入口/统计/权限扩展
  refactor → 沿用已有产品业务定义，但现有方案存在架构、性能、体验、稳定性或规则实现问题
  new → 业务目标、用户链路、核心对象或价值主张无法合理挂接到已有产品
  high 匹配下仍选择 new 时，提示重复建设风险并记录用户理由到 decision-log.md
```

## 9. 结果呈现格式

### high/medium 匹配

```
🔍 产品匹配结果

产品：{name}（{id}）
匹配度：⭐⭐⭐ 高度匹配 / ⭐⭐ 部分匹配
总分：{score}/100

维度匹配详情：
  ✅ 问题本质（100）：都是"..."
  ✅ 痛点（75）：...
  ⚠️ 用户角色（50）：已有产品面向"..."，你的需求涉及"..."，可作为角色扩展
  ✅ 业务领域（75）：同属...领域

已有产品解读卡：
  它解决的问题：...
  覆盖的业务闭环：...
  已有能力：...
  和你需求可能重叠的点：...
  还需要确认的点：...

下一步：进入需求分析 intake 的复用引导，围绕你的业务目标继续核对关键事实，系统据此判断覆盖点、差异点和扩展方式。

业务事实核对示例：
  你的需求是「{用户需求复述}」。
  我注意到已有产品已经覆盖「{已有产品业务语言解释}」。我的初步判断是：可以先尝试复用「{已有能力}」，差异可能落在「{扩展方向}」。
  本轮我先核对一个业务事实：这次自动校验主要发生在哪个业务时点？
  A. 入网交维/资源入库前，数据提交时自动校验
  B. 日常资源维护中，资源变更后自动校验
  C. 定期盘点/巡检时，批量校验存量数据
  D. 跨系统对账时，比较多个系统中的资源数据
  E. 补充描述：我自己填写
  F. 强制跳过：这个问题暂时不回答，记录为待验证并继续
```

### 无匹配

```
🔍 产品匹配结果

未找到与你的需求匹配的已有产品。建议项目类型为 new，等待用户确认。
产品库中现有产品：
  - {name}（{id}）
```

## 10. 边界情况

| 情况 | 处理 |
|------|------|
| 产品库为空 | 匹配结果 none，建议项目类型为 new，等待用户确认，并提示后续可用 `export-to-library.sh` 导出 |
| 用户描述过于模糊 | 所有维度评分 0-25，提示补充细节，提供"补充后重新匹配/暂按 new 建议继续"选项 |
| 产品库文档不完整 | 缺失维度评分为 0，标注"缺少{文档类型}，匹配可能不准确" |
| 需求跨多个产品 | 全部呈现按总分排序，提示"可能需分别处理"，让用户选一个为主关联 |
| 用户认为应匹配但未匹配 | 提供"手动指定关联产品"选项，展示所有产品列表 |

## 11. 准确度保障

- 所有维度比对都是语义比对，AI 必须理解概念语义含义
- 每个维度评分必须附带判断理由（提取了什么信息、对应了什么内容、为何判定该等级）
- 匹配结果用于形成候选关联、已有产品解读卡、覆盖点、差异点和项目类型建议，最终选择由用户确认
- 用户可手动指定关联产品或覆盖任何匹配结果
- 所有覆盖决策记录到 `decision-log.md`
