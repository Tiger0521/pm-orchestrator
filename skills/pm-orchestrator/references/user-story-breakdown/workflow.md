# 需求拆解执行流程

本文件在 `mode=draft` 执行 user-story-breakdown 时读取，定义 9 步工作流、上游质量门和项目类型自适应规则。

## 总体流程

当前工作流分为 9 步，按两段执行。第一段读取上游、梳理角色规则并拆分主干 Story；第二段补充异常分支、编写 GWT 并生成溯源矩阵。

第 2、3、4、5、6、7 步需要用户确认。确认方法见 `confirmation-method.md`。

## 第一段：读取上游 + 主干拆分

### 第 1 步：读取上游文档

- 读取 `refs.json`，筛选 Epic 和 Feature 节点。
- 读取所有 Feature 和 Epic 文档的 frontmatter 和正文，尤其关注业务流程、业务规则、用户角色、能力描述、能力目标字段。
- 检查上游质量门（见下方“上游质量门”），不通过时返回 `needs-input`。

### 第 2 步：梳理角色和规则

- 从 Epic 用户角色表和 Feature 用户角色字段中汇总所有角色。
- 从 Feature 业务规则字段中汇总所有业务规则。
- 从 Feature 业务流程字段中汇总所有业务流程步骤。
- 输出“角色-规则-流程”摘要回执，按确认流程向用户确认（先展示摘要，再给理解回执，最后提聚焦问题）。

### 第 3 步：按角色拆分主干 Story

- 基于已确认的角色和 Feature 能力，按角色识别主干目标。
- 每个 Story 遵循三段式格式：“作为 [角色]，我想要 [目标]，以便于 [价值]”。
- 按拆分维度（角色 -> 流程步骤 -> 操作类型）组织 Story。
- 识别哪些 Feature 需要补充体验型 Story。
- 对每条 Story 做 INVEST 检查，不通过的调整颗粒度后重新输出。
- 输出主干 Story 草稿清单，按确认流程向用户确认拆分方案。

## 第二段：异常分支 + GWT + 溯源矩阵

### 第 4 步：枚举异常分支

- 为主干 Story 补充异常场景（权限不足、数据为空/超限、网络异常/超时、并发冲突、重复提交、输入格式错误）。
- 每个异常场景判断是否在该 Feature 的业务规则中可能发生；不可能发生的跳过并说明理由。
- 异常场景作为独立 AC 或独立 Story（视颗粒度决定）。
- 输出异常分支补充清单，按确认流程向用户确认。

### 第 5 步：编写 GWT 验收标准

- 为每条 Story 编写 3-8 条 GWT 验收标准。
- 覆盖正常路径 + 异常路径 + 边界场景。
- 检查 GWT 格式规范（Given 前置状态、When 触发动作、Then 期望结果）。
- 输出完整 Story 落盘预览（含三段式 + 优先级 + Story Points + 全部 AC），按确认流程向用户确认。

### 第 6 步：优先级排序与 Story Points 估算

- 标注每条 Story 的优先级（P0/P1/P2）。
- 给出 Story Points 估算建议（1/2/3/5/8/13），附注“建议值，待团队确认”。
- 检查估算超过 13 的 Story 是否需要拆分。
- 输出优先级和估算清单，按确认流程向用户确认。

### 第 7 步：生成溯源矩阵

- 建立 Story 与 Feature 的追溯关系。
- 检查覆盖度：每个 Feature 至少被一条 Story 实现；所有 P0 Feature 已覆盖。
- 输出溯源矩阵草稿，按确认流程向用户确认。

### 第 8 步：用户确认

- 用户确认 Story 拆解方案、优先级、Story Points 和 GWT 验收标准。
- 用户确认溯源矩阵覆盖度。
- 用户确认前，不进入落盘。

### 第 9 步：落盘

- 当 `mode=persist` 且用户已确认时，读取 `persist-guide.md` 执行落盘流程。
- 不用 AI 逐行 Write：Story 和溯源矩阵的 Markdown 文件由脚本渲染写入，AI 只负责准备结构化数据。

## 上游质量门

拆解前检查上游文档质量：

- Feature 的业务流程字段非空，含关键步骤和触发条件。
- Feature 的业务规则字段非空，含异常处理。
- Epic 的用户角色字段非空，含角色诉求。
- 如果上游文档存在字段缺失或内容空洞，向主调度器返回 `needs-input`，附注缺失字段清单。
- 检查上游 Feature 的 `status` 是否为 `approved` 或 `review`；若有 Feature 仍为 `draft`，向主调度器报告 `needs-input`。

## 规模自适应

项目类型在需求分析 intake 中收敛，写入 `progress.json` 的 `projectType` 字段。需求拆解阶段读取该字段决定流程深度：

| 环节 | 全新项目 (`new`) | 迭代项目 (`iteration`) | 重构项目 (`refactor`) |
| ---- | -------- | -------- | -------- |
| 上游读取 | 读取全部 Feature + Epic | 读取新增 Feature + 关联 Epic | 读取全部 Feature + Epic |
| 角色梳理 | 完整列出所有角色 | 聚焦新增能力涉及的角色 | 聚焦受影响角色和权限变化 |
| Story 拆分 | 完整拆分全部 Feature | 聚焦新增 Feature 的 Story 拆分 | 只产出非功能性需求的 Story（性能、安全、兼容性等） |
| 异常枚举 | 完整枚举 | 聚焦新增能力的异常场景 | 聚焦重构引入的新异常场景 |
| 覆盖度检查 | 全部 Feature 覆盖 | 新增 Feature 覆盖 | 已有 Feature 覆盖 |

路由不是机械省略。若拆解过程中暴露上游 Feature 信息不足，返回主调度器要求补问或回退。
