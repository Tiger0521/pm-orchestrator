# User Story 写作规范

本文件定义 User Story 三段式描述和 GWT 验收标准的写作质量基线。`instruction.md` 引用本文件作为拆解时的写作标准，`checklist.md` 的 Story 质量和验收标准质量校验项引用本文件作为判断依据，`validate-story.sh` 的校验规则派生自本文件。

---

## 五条通用规律

以下五条适用于 User Story 的所有文字内容（三段式 + GWT 验收标准）。任一条不满足，视为文字质量不达标。

### 1. 总结先行

三段式的"以便于"是本 Story 的价值总结，不是功能描述的重复。验收标准的第一条应概括正常路径的核心结果。

- **差**：价值写"提升用户体验"，和活动"查看列表"几乎一样
- **好**：价值写"快速定位需要修改的配置版本，减少人工翻页时间"

### 2. 加粗关键词领条

验收标准每条以 `**加粗关键词**` 开头，关键词是场景判断词，不是"AC1"等编号。

- **差关键词**：`AC1`、`场景一`、`正常`（太泛或无信息）
- **好关键词**：`**正常提交**`、`**权限不足**`、`**空列表**`、`**重复名称**`（有判断、有场景）

### 3. 具名细节

角色用具体名称（"VIP 会员"而非"用户"），GWT 的 Then 写具体提示语或预期数据（"系统提示'配置名称已存在'"而非"系统提示错误"）。

- **差**：`Then 系统提示错误`
- **好**：`Then 系统提示"配置名称已存在"并阻止提交`

### 4. 结论跟"为什么"

三段式的"以便于"必须说清价值——"更快完成支付"是合格，"提升体验"是不合格。价值描述和活动描述几乎一样时，说明这条 Story 没有想清楚用户目标。

- **差**：`以便于查看列表`
- **好**：`以便于快速定位需要修改的配置版本，减少人工翻页时间`

### 5. 视觉结构匹配内容

验收标准用分条列表，每个 AC 一个场景；多角色对比用表格；不压成流水句。

| 内容形态 | 视觉结构 |
| --- | --- |
| 多条验收标准 | 加粗关键词领条的编号列表 |
| 角色权限对比 | 表格 |
| 流程步骤 | 流程图或分条 |

## 三段式写作规范

每条 User Story 必须采用标准三段式格式：

```
作为 [角色]，我想要 [目标]，以便于 [价值]
```

### 角色（Who）

角色必须具体到角色名，不使用笼统的"用户""管理员"。角色来源可追溯到 Epic 的用户角色表或 Feature 的用户角色字段。如果拆解过程中发现 Feature 未覆盖的角色，先追问用户确认是否为真实角色，再写入 Story。

| 合格 | 不合格 |
| --- | --- |
| 作为 **VIP 会员**，我想要保存我的支付信息，以便于下次结账时更快捷地完成支付 | 作为 **用户**，我想要保存信息，以便于下次方便 |
| 作为 **系统管理员**，我想要查看模型配置列表，以便于快速定位需要修改的配置版本 | 作为 **管理员**，我想要查看列表，以便于管理 |
| 作为 **新注册用户**，我想要重置密码，以便于在忘记密码时恢复账号访问 | 作为 **普通人**，我想要改密码，以便于安全 |

### 活动（What）

活动描述用户意图，不是技术实现。不要写接口名、页面名、模块名。如果上游 Feature 的业务流程是"用户提交配置 → 系统校验 → 保存到数据库"，Story 的活动是"提交配置并查看校验结果"，不是"调用校验接口并写入数据库"。

| 合格 | 不合格 |
| --- | --- |
| 我想要 **保存我的支付信息** | 我想要 **调用 POST /api/payment** |
| 我想要 **查看模型配置列表** | 我想要 **打开配置管理页面** |
| 我想要 **批量导入配置** | 我想要 **执行 batch-import.sql 脚本** |

### 价值（Why）

价值必须具体、可感知。如果价值描述和活动描述几乎一样（如"以便于查看列表"），说明这条 Story 没有想清楚用户目标，需要重新梳理。

| 合格 | 不合格 |
| --- | --- |
| 以便于 **下次结账时更快捷地完成支付** | 以便于 **提升用户体验** |
| 以便于 **快速定位需要修改的配置版本，减少人工翻页时间** | 以便于 **提高效率** |
| 以便于 **在忘记密码时无需联系客服即可恢复账号访问** | 以便于 **方便管理** |

## GWT 验收标准写作规范

每条 Story 必须有 3-8 条验收标准，采用 Given-When-Then 格式。每条 AC 以 `**加粗关键词**` 开头，后接 GWT 内容。

**AC 格式**：

```
1. **正常提交**：Given 算法工程师已登录系统，When 填写模型名称、版本、参数并点击提交，Then 系统创建配置并返回成功提示"创建成功"
2. **必填项校验**：Given 算法工程师未填写模型名称，When 点击提交按钮，Then 系统提示"模型名称不能为空"并阻止提交
```

### Given（前置状态）

Given 描述前置状态，不是操作步骤。不要写"先创建一个配置"——这是操作步骤，不是状态。

| 合格 | 不合格 |
| --- | --- |
| Given **已存在名称为"v1"的模型配置** | Given **配置已存在** |
| Given **用户已登录且有管理员权限** | Given **用户有权限** |
| Given **配置列表为空** | Given **没有数据** |

### When（触发动作）

When 描述具体的触发动作，是用户操作或系统事件。不写"系统执行"这种笼统描述。

| 合格 | 不合格 |
| --- | --- |
| When **用户点击提交按钮** | When **用户提交表单** |
| When **定时任务触发** | When **系统自动运行** |
| When **用户在名称框输入已存在的配置名"v1"并提交** | When **用户输入重复数据** |

### Then（期望结果）

Then 写具体的系统行为，包括提示语、页面变化、数据变化。提示语用引号标出，数据变化写明具体字段。

| 合格 | 不合格 |
| --- | --- |
| Then **系统提示"配置名称已存在"并阻止提交** | Then **系统提示错误** |
| Then **列表展示空状态提示"暂无配置，点击创建第一条"** | Then **列表显示空状态** |
| Then **系统创建配置记录，状态字段标记为"启用"，列表刷新后展示该记录** | Then **系统创建成功** |

### 每条 AC 只覆盖一个场景

不要把"正常提交成功"和"名称为空提示错误"塞进同一条 AC。每条 AC 独立覆盖一个场景，加粗关键词即场景判断词。

### 验收标准分类

每条 Story 的 AC 必须覆盖三类路径：

| 类型 | 说明 | 示例 |
| --- | --- | --- |
| 正常路径 | 用户按预期流程操作，系统返回成功 | 填写完整信息并提交，系统创建配置并返回成功 |
| 异常路径 | 权限不足、数据为空/超限、网络异常、并发冲突、重复提交、输入格式错误 | 提交空表单，系统提示必填项不能为空 |
| 边界场景 | 数据边界、状态边界、权限边界 | 列表为空时展示空状态提示 |

### 异常路径枚举清单

对每条主干 Story，至少考虑以下异常场景：

- **权限不足**：非授权角色尝试操作
- **数据为空/超限**：必填项为空、输入超出长度或数量限制
- **网络异常/超时**：请求未到达或响应超时
- **并发冲突**：多人同时编辑同一对象
- **重复提交**：短时间内多次点击提交
- **输入格式错误**：邮箱格式、数值范围、特殊字符

枚举不是机械遍历。如果某个异常场景在该 Feature 的业务规则中不可能发生（如内部工具无网络场景），可以跳过但需说明理由。

## 落盘前自检清单

输出 `draft-ready` 或执行 `persist` 前，逐条 Story 逐条 AC 快速扫描。任一条不满足，重写该 Story 或 AC 后再输出。不要带着格式问题进入草稿预览。

### 三段式自检

- [ ] 角色是否具体到角色名（不是"用户""管理员"）？
- [ ] 角色来源是否可追溯到 Epic 或 Feature 的用户角色字段？
- [ ] 活动是否描述用户意图（不是技术实现、接口名、页面名）？
- [ ] 价值是否具体可感知（不是"提升体验""提高效率"等空泛词）？
- [ ] 价值是否和活动有区分度（不是活动的重复）？

### 验收标准自检

- [ ] 每条 AC 是否以 `**加粗关键词**` 开头（不是"AC1"等编号）？
- [ ] 加粗关键词是否是场景判断词（不是"正常""异常"等泛词）？
- [ ] Given 是否描述前置状态（不是操作步骤）？
- [ ] When 是否描述具体触发动作（不是"系统执行"等笼统描述）？
- [ ] Then 是否写具体系统行为（不是"提示错误"等笼统描述）？
- [ ] 提示语是否用引号标出（如"配置名称已存在"）？
- [ ] 数据变化是否写明具体字段？
- [ ] 每条 AC 是否只覆盖一个场景（没有多条件多结果塞进一条）？
- [ ] AC 数量是否在 3-8 条范围内？
- [ ] 是否覆盖正常路径 + 异常路径 + 边界场景三类？
- [ ] 跳过的异常场景是否已说明理由？

### 视觉结构自检

- [ ] 该用表格的地方用了表格（如多角色对比）？
- [ ] 该用分条的地方用了分条（如多条 AC）？
- [ ] 没有一段流水句压到底？
