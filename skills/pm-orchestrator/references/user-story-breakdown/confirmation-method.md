# 提问与确认方法

本文件在 `mode=draft` 需要与用户确认拆解决策时读取。主 agent 只负责通用交互协议（每轮一个问题、选项格式、补充描述和跳过），本文件定义 user-story-breakdown 阶段的确认方法。

## 理解回执

每次用户回答后、下一轮确认或提问前，必须先用 2-5 行反馈当前理解：

```text
已确认：
- 角色：VIP 会员、系统管理员
- 主干 Story：创建配置、查看配置列表、编辑配置
- 异常场景：已覆盖权限不足、名称重复、连接超时
待确认：是否需要补充“批量导入配置”的体验型 Story
```

回执不是第二个问题，不得夹带新的追问。回执的目的是让用户在下一轮确认前有机会纠偏。如果理解有误，用户先纠偏，再继续。

## 确认流程

9 步工作流中，第 2、3、4、5、6、7 步各需要用户确认一次。确认不是简单问“对吗？”，而是：

1. 先展示结构化产出（角色-规则-流程摘要 / Story 草稿清单 / 异常分支清单 / 完整 Story 落盘预览 / 优先级和估算清单 / 溯源矩阵），每条用加粗关键词领条。
2. 再给出理解回执（已确认了什么、待确认什么）。
3. 最后提出一个聚焦问题（选择题或确认题），遵守每轮一个问题原则。

确认问题示例：

| 步骤 | 确认问题 |
| ---- | -------- |
| 第 2 步（角色规则） | “以上角色和规则覆盖了本次拆解的全部用户场景，还是需要补充？” |
| 第 3 步（主干 Story） | “以上 5 条主干 Story 的拆分方案是否符合预期？需要调整或合并哪些？” |
| 第 4 步（异常分支） | “以上异常场景是否有遗漏？是否有不应覆盖的场景需要移除？” |
| 第 5 步（GWT） | “以上 GWT 验收标准是否覆盖了全部关键路径？有没有标准过严或过松的？” |
| 第 6 步（优先级） | “以上优先级排序是否反映业务紧急度？SP 估算是否需要调整？” |
| 第 7 步（溯源矩阵） | “以上溯源矩阵覆盖度是否完整？有没有 Feature 被遗漏？” |

不得问“对吗？”：确认问题必须有具体指向，例如“需要调整或合并哪些？”“有没有遗漏？”“标准是否过严或过松？”。不要问“对吗？”“可以吗？”“有没有问题？”这种无指向的确认。

确认后翻状态：用户确认的内容从 `pending` 翻转为 `confirmed`（见 `output-contract.md` 的草稿状态追踪）。用户提出修改的内容保持 `pending` 并记录修改方向，下一轮只问修改相关的问题。

## 问题选择优先级

如果同一轮有多个待确认项，按以下优先级选择只问一个：

1. 影响后续所有拆解的基础决策（角色、范围）。
2. 影响 Story 数量和颗粒度的拆分方案。
3. 影响验收标准的异常场景。
4. 优先级和估算。
5. 溯源矩阵覆盖度。

## 范围漂移防护

如果用户在确认过程中提出新的角色、新的异常场景或新的 Feature 范围，必须判断是否属于当前拆解范围：

- 新角色在已有 Feature 的用户角色字段中：纳入，补充对应 Story。
- 新异常场景在已有 Feature 的业务规则中：纳入，补充对应 AC。
- 新角色或新场景不在已有 Feature 中：记录到 `tracking-log.md`，标记为“超出当前拆解范围，需回到需求分析阶段补充 Feature”，本次不展开。
