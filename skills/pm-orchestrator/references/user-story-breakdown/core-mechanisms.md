# 需求拆解核心机制

本文件在 `mode=draft` 执行拆解时读取，定义 User Story 拆分、GWT 编写和质量判断的核心原则。

## INVEST 检查

每条 User Story 都应满足 INVEST 六原则：

| 原则 | 含义 | 检查问题 |
| ---- | ---- | -------- |
| Independent | 独立 | 这条 Story 能独立交付和验证吗？ |
| Negotiable | 可协商 | 实现细节是否留有余地？ |
| Valuable | 有价值 | 对用户或业务有明确价值吗？ |
| Estimable | 可估算 | 团队能判断工作量吗？ |
| Small | 足够小 | 一个 Sprint 内能完成吗？ |
| Testable | 可测试 | 有明确的验收标准吗？ |

检查时机：每条 Story 输出草稿前逐条检查。任一原则不满足时，必须调整 Story 颗粒度（拆分或合并）后重新输出，不得带着 INVEST 缺陷进入草稿预览。

独立性问题处理：如果两条 Story 存在强依赖（A 必须先于 B 完成），尝试重新拆分边界使其独立。无法独立时，在 Story 中显式标注依赖关系并记录到 `tracking-log.md`，同时在溯源矩阵的映射关系中标注。

## 三段式格式

每条 User Story 必须采用标准三段式格式：

```text
作为 [角色]，我想要 [目标]，以便于 [价值]
```

角色来源：角色从 Epic 的用户角色表和 Feature 的用户角色字段中提取，不自行编造。如果拆解过程中发现 Feature 未覆盖的角色，先追问用户确认是否为真实角色，再写入 Story。

活动描述边界：活动描述用户要做的事，不描述系统怎么做。“查看模型配置列表”是合格的活动描述；“调用 GET /api/configs 接口返回配置数据”不合格，因为它描述了技术实现。

三段式写作规范详见 `writing-paradigm/user-story-writing.md`。

## GWT 验收标准

每条 Story 必须有 3-8 条验收标准，采用 Given-When-Then 格式：

```text
1. **正常提交**：Given [前置条件]，When [用户操作/系统事件]，Then [期望结果]
```

- Given 描述前置状态，不是操作步骤。
- When 描述触发动作，是用户操作或系统事件。
- Then 描述期望结果，是可验证的系统行为。
- 每条 AC 只覆盖一个场景，不把多个条件和结果塞进同一条。
- 每条 AC 以 `**加粗关键词**` 开头，关键词是场景判断词，不是“AC1”等编号。

验收标准分类：

| 类型 | 说明 | 示例 |
| ---- | ---- | ---- |
| 正常路径 | 用户按预期流程操作，系统返回成功 | 填写完整信息并提交，系统创建配置并返回成功 |
| 异常路径 | 权限不足、数据为空/超限、网络异常、并发冲突、重复提交、输入格式错误 | 提交空表单，系统提示必填项不能为空 |
| 边界场景 | 数据边界、状态边界、权限边界 | 列表为空时展示空状态提示 |

GWT 写作规范详见 `writing-paradigm/user-story-writing.md`。

## 异常分支枚举

对每条主干 Story，至少考虑以下异常场景：

- 权限不足
- 数据为空/超限
- 网络异常/超时
- 并发冲突
- 重复提交
- 输入格式错误

枚举不是机械遍历。如果某个异常场景在该 Feature 的业务规则中不可能发生（如内部工具无网络场景），可以跳过但需说明理由。

## 颗粒度控制

Story 颗粒度直接影响估算准确性和交付节奏。

拆分信号：

- 角色重叠：一条 Story 覆盖多个角色的不同目标。
- 流程重叠：一条 Story 覆盖多个独立流程步骤。
- SP > 8：估算超过 8 个 Story Points。
- AC > 8：验收标准超过 8 条。

合并信号：

- 同数据对象：多条 Story 操作同一数据对象的同一生命周期。
- SP = 1 且不独立：极小 Story 且依赖其他 Story。
- AC 重叠：多条 Story 的验收标准高度重叠。

拆分维度按优先级为：角色 > 流程步骤 > 操作类型。禁止按技术层拆分（前端/后端/数据库）。

机制边界：颗粒度判断的最终裁决者是用户，不是 agent。agent 按信号给出拆分/合并建议，用户确认后执行。SP > 13 的 Story 必须拆分才能落盘；SP = 1 的 Story 可以落盘但需标注“建议评估是否合并”。

## 优先级与估算

优先级继承自 Feature 的优先级，可在同 Feature 内按业务紧急度调整。估算使用斐波那契数列（1/2/3/5/8/13），依据来自 Feature 的资源投入字段和业务流程复杂度。

优先级和 Story Points 均为建议值，最终由开发团队确认。agent 输出时附注“建议值，待团队确认”。SP > 13 的 Story 必须拆分才能落盘。

## 反谄媚与前提挑战

遇到模糊、方案先行或证据不足的用户反馈时，禁止使用附和性表达，改用追问方式。

| 用户表达 | 不接受的弱回应 | 应追问的方向 |
| -------- | -------------- | ------------ |
| “全部拆出来” | “可以全部拆分” | 哪些是 P0 必须的？哪些可以后续迭代？ |
| “按页面拆” | “按页面拆分很合理” | 用户在每个页面要完成什么业务目标？页面只是入口，不是拆分依据 |
| “这条很简单” | “确实简单” | 简单到什么程度？有没有异常场景？权限有没有差异？ |
| “加个体验优化” | “体验优化很重要” | 哪个具体场景、哪个操作行为代表体验变好？验收标准怎么写？ |
| “这条不用写异常” | “可以跳过异常” | 这个场景是否可能出现权限不足、数据为空或并发？如果不可能，理由是什么？ |

规则：可以语气礼貌，但结论必须有立场；不能用“可以考虑、可能、建议关注”掩盖拆解缺口。
