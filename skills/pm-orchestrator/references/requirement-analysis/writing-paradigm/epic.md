# Epic 字段范式

> 前置读取：本文件依赖 `writing-paradigm/general-rules.md` 中的六条通用规律和六种语言范式定义。生成 Epic 草稿前，先读 `general-rules.md`，再读本文件。
>
> **素材来源**：撰写每个字段的最终润色值时，必须从字段 JSON 的 `qa_log` 中读取该字段的全部 Q&A 记录作为素材。`qa_log` 里的用户回答是原始事实来源，最终润色值是基于这些事实按范式写出的产物。只能比 Q&A 内容多（补充因果、结构化），不能比 Q&A 内容少（丢信息）。

## 各字段详解与好差对比

### 需求背景

**写法**：引用需求卡片，自动填入。格式为"本 Epic 派生自 [@req-xxx]：{{需求卡片标题和一句话结论}}"。无需追问，不套范式。

---

### 产品名称

**写法**：业务语言命名，能让非项目成员看懂"这件事解决什么业务结果"。不用技术代号或模块名。一句话。

- **差**：`res-lifecycle-sys`
- **好**：`网络资源全生命周期系统`

---

### 产品定位

**范式**：B（blockquote 核心论断 + 分要点展开）

**规则**：
1. 核心论断用 blockquote + 加粗，是整个字段的"题眼"
2. 论断后用"这句话有 N 个要点"过渡
3. 每个要点以加粗关键词开头，后接句号和解释段落
4. 解释段落要有"不是…而是…"的对比论证

**差写法**：

```
配置驱动的资源全生命周期管控平台，在网络资源数据进入网络资源中心的源头设立数据质量关卡。三个要点：1. 本质是数据治理，而非工单工具--核心价值是让准确完整的资源数据真正沉淀到资源中心，靠'卡住数据'而非'加重人'；2. 配置驱动，而非一事一开发--资源对象、属性、校验规则、流程、外部接口均可配置；3. 平台化，多场景统一承载--入网交维是首发场景，以统一工单模型承载多类业务场景
```

> 差在哪：核心论断没有用 blockquote 突出；三个要点压成一段；要点没有独立成条；缺少"不是…而是…"的对比论证展开。

**好写法**：

```
> **网络资源全生命周期系统，是一个配置驱动的资源全生命周期管控平台，在网络资源数据进入网络资源中心的源头设立数据质量关卡。**

这句话有三个要点：

1. **本质是数据治理，而非工单工具**。系统的核心价值是让"准确、完整的资源数据真正沉淀到资源中心"，而不是单纯"让人填工单"。质量关卡设在**数据**上--数据必须通过校验才能进入资源中心，这道闸门不松动；但管控的方式是**嵌入一线生产、挂接业务事件、最小化一线动作**，而非要求一线在系统里另走一遍与实际脱节的流程。把数据质量管控从事后清查前移到源头，靠的是"卡住数据"而非"加重人"。
2. **配置驱动，而非一事一开发**。资源对象、属性、校验规则、流程、外部接口均可配置，业务变化由运营团队配置应对，而非依赖厂家代码开发。
3. **平台化，多场景统一承载**。入网交维只是首发场景，系统以统一的工单模型承载入网、关停、退网、割接、租赁费等多类业务场景，可持续扩展。
```

### 产品目标

**范式**：A（总结开头 + 分条列点）

**规则**：
1. 第一句总结目标的方向（如"沿三条主线展开"）
2. 每条以加粗关键词开头，含"从X到Y"的量化变化
3. 至少 2 条，每条独立成段

**差写法**：

```
1. 数据质量从事后清查前移到源头管控：变'先污染后治理'为'源头把关'，质量关卡设在数据进入资源中心前，校验通过率、关键字段完整率首次可统计可考核；2. 防重复投资：让建成的资源'数据上也存在'，通过线上交维+数据校验对齐'建成'与'数据上存在'，堵住重复投资漏洞；3. 流程规范化：把管理要求嵌进生产环节，线上替代线下，审批按多维路由
```

> 差在哪：没有总结开头；三条压成一段；缺少量化目标值。

**好写法**：

```
产品目标沿三条主线展开，每条都对应一个可验证的业务结果变化：

- **数据质量从事后清查前移到源头管控**：变"先污染后治理"为"源头把关"，质量关卡设在数据进入资源中心前。校验通过率、关键字段完整率首次可统计、可考核。
- **防重复投资**：让建成的资源"数据上也存在"，通过线上交维 + 数据校验对齐"建成"与"数据上存在"，堵住因数据不及时入库导致的重复建设漏洞。
- **流程规范化**：把管理要求嵌进生产环节，交维从钉钉/邮件/纸质签字搬到线上，审批按区域/专业/网络层级/金额多维路由，落实集团全流程贯通要求。
```

### 用户角色

**范式**：C（开头定位 + 表格）

**规则**：
1. 表格上方一句话说明覆盖的角色范围和管理层级
2. 表格至少 3 列：角色、所在环节/职责、核心诉求
3. 核心诉求列要具体，不能只写"提升效率"

**差写法**：

```
一线管线专业人员（施工/维护的一线操作人员）、建设方/外包厂家（负责施工和资源数据录入）、维护方/运维团队（交维后使用资源数据，负责日常运维和故障排查）、集团副总（自上而下推动全流程贯通与规范）、中心网络治理工作组（制定年度治理方案）
```

> 差在哪：逗号拼接的流水句；没有表格；没有"所在环节"和"核心诉求"的结构化拆分。

**好写法**：

```
系统服务于网络资源入网交维链条上的多类角色，覆盖集团、省公司、地市多个管理层级：

| 角色 | 所在环节 | 核心诉求 |
| --- | --- | --- |
| 一线管线/维护人员 | 交维受理、资源核对 | 集中看到交维任务与状态，减少人工上报与核对损耗 |
| 建设方/外包厂家 | 施工、资源数据录入 | 按规范录入资源数据，顺利完成交维 |
| 专业主管 | 数据/流程配置、审批 | 自主配置本专业资源对象与校验规则，把控数据质量 |
| 各级管理部门 | 流程审批、监督 | 按管理层级分支流程，督促建设质量 |
| 运营团队 | 配置维护、系统运营 | 通过配置快速响应一线需求，业务规则变更不必等研发排期 |
| 上游/下游系统 | 系统对接 | 智能规建系统推送交维数据，网络资源中心接收归档数据 |
```

### 核心场景

**范式**：D（开头定位 + 场景描述/流程图 + 特征）

**规则**：
1. 开头一句话说明场景覆盖的业务范围
2. 每个核心场景用"场景名 + 用户从什么状态开始 + 经过什么动作 + 到达什么结果"描述
3. 如果场景有流转关系，用 mermaid 或 text 流程图
4. 场景至少 1 个，每个场景独立成段

**差写法**：

```
光缆入网交维（建设方施工完成后，将光缆资源数据提交给维护方验收，完成从建设到运维的交接）、哑资源数据校验（光缆等无源资源与网管采集数据自动比对，验证资源数据的准确性和完整性）、资源数据录入、多专业数据贯通
```

> 差在哪：括号内的描述压缩成一句；多个场景用顿号拼接；没有"用户从什么状态到什么状态"的流转；没有流程图。

**好写法**：

```
系统以入网交维为首发场景，同时承载关停、退网、割接等多类业务场景。以下是最核心的场景：

**场景一：光缆新建入网交维**

建设方施工完成后，将光缆资源数据提交给维护方验收，完成从建设到运维的交接：

```text
智能规建推送交维数据 -> 系统生成入网交维工单 -> 按专业/场景/网络层级路由审批流程 -> 多级审批 -> 配置化数据校验 -> 校验通过则归档并推送资源中心；不通过则退回补充
```

**场景二：哑资源数据校验**

光缆等无源资源缺乏与网管采集数据的有效比对手段。系统在交维归档时自动执行"查询 + 比对 + 阈值"校验，验证资源数据的准确性和完整性，不合格数据被拦在源头之外。

**场景三：多专业数据贯通**

光缆、传输、动环、无线四个专业之间实现系统对接和数据流转。上游推送的交维数据按专业路由到对应审批流程，校验通过后统一归档到网络资源中心。
```

### 产品价值

**范式**：A（总结开头 + 分条列点）

**规则**：
1. 第一句总结价值的方向（如"沿四条主线展开"）
2. 每条以加粗关键词开头，含"变X为Y"的变化和不做会损失什么
3. 至少 2 条，每条独立成段

**差写法**：

```
1. 数据质量：从事后清查前移到源头管控，变'人情签字'为'机制把关'；2. 防重复投资：让建成与数据存在对齐；3. 流程规范化：线上全程可追溯；4. 平台化扩展：一次建设多场景复用
```

> 差在哪：没有总结开头；每条只有半句话；缺少"不做会损失什么"；缺少具名细节。

**好写法**：

```
系统的价值不是"上线一个工单工具"，而是改变网络资源数据从产生到入库的管控方式。收益沿四条主线展开：

- **数据质量：从事后清查前移到源头管控**：变"先污染后治理"为"源头把关"，质量关卡设在数据进入资源中心前。变"人情签字"为"机制把关"，用配置化校验替代人工核对。校验通过率、关键字段完整率第一次变成可统计、可考核的指标。
- **防重复投资**：让建成的资源"数据上也存在"。政企业务开通以资源数据为依据做核查，资源建成了但数据没及时入库，再次核查时仍判定"资源不具备"，于是重复建设、重复拉缆。系统通过源头管控堵住这个漏洞。
- **流程规范化**：交维从钉钉、邮件、纸质签字搬到线上，全程可追溯、可统计、可监督。审批按区域、专业、网络层级、金额多维路由，各级管理部门在自己的环节履职。
- **平台化扩展**：一次建设，多场景持续复用。新增专业或场景主要靠配置而非重新开发，业务规则维护权交还运营团队，从根本上摆脱对外包研发力量的依赖。
```

### 产品范围与边界

**范式**：A（范围内分条 + 范围外分条带"为什么不"理由）

**规则**：
1. 范围内：分条列出包含的内容，每条以加粗关键词开头
2. 范围外：分条列出不做的内容，每条以 `**不做XX**：因为…` 的结构写明理由
3. 范围外的理由要回答"为什么不做"，不能只写"不在范围内"

**差写法**：

```
范围内：光缆、传输、动环、无线四个专业同时覆盖，七个能力在四个专业同时落地，不区分试点
范围外：存量系统迁移、另一个资源库、替代数据治理平台、网络隐患治理
```

> 差在哪：范围内只有一句话压缩；范围外只有标签没有理由；没有加粗关键词。

**好写法**：

**范围内**：

```
- **四专业入网交维流程**：光缆、传输、动环、无线的新建资源入网，覆盖一干/二干/本地多网络层级
- **配置驱动的数据质量关卡**：资源对象、属性、校验规则、流程、接口均可配置
- **与上下游系统的接口对接**：智能规建系统（上游推送）、网络资源中心（下游归档）、能力平台等
- **多场景统一承载**：入网交维、无线基站暂时关停、租赁费审批等复用同一套工单与流程体系
```

**范围外**：

```
- **不做存量系统迁移**：与现有厂家系统隔离运行，本期不触碰、不迁移现网功能。原因是现有系统存在严重的组织与技术双重债务，迁移成本高且收益不明确。
- **不是又一个资源库**：本系统不存储、不接管资源主数据的归属，资源数据的 SSOT 仍在网络资源中心。本系统只在数据入库前设关卡、做贯通，与既有系统是互补而非重叠。
- **不替代数据治理平台**：数据治理平台负责存量数据的事后清查，本系统负责新入网数据的源头管控，两者互补。
- **不纳入网络隐患治理**：网络隐患是独立系统，与本系统无直接关系，不在本范围内。
```

### 建设思路

**范式**：设计理念范式（不是优先级排序，是设计主张的展开）

建设思路是系统的**设计理念**--系统建立在什么设计主张之上，这些主张决定了系统能否长期低成本演进。不是功能清单，不是 P0/P1/P2 排序，不是开发计划。

**规则**：
1. 开头一句话总结：系统不是功能的简单堆砌，而是建立在 N 个设计主张之上
2. 每个主张用 `### 序号. 主张名称：一句话概括` 三级标题领起，独立成节，有序号便于引用
3. 每个主张内有：展开说明（可分条，每条加粗关键词领起）+ `**这一理念的价值**` 段落
4. 可选 blockquote 标注诚实边界（哪些已验证、哪些是方向而非承诺）
5. 至少 2 个设计主张

**差写法**：

```
| 优先级 | 建设内容 | 排序依据 |
| --- | --- | --- |
| P0 | 光缆入网交维、配置化校验引擎 | 先跑通光缆交维最小闭环 |
| P1 | 工单管理、审批流程 | 用统一工单模型承载其余场景 |
| P2/暂缓 | 定时任务调度、业务日志查询 | 运营辅助能力 |
```

> 差在哪：这是优先级排序，不是设计理念。没有说清系统建立在什么设计主张之上，没有展开每个主张的内涵和价值。

**好写法**：

```
系统不是功能的简单堆砌，而是建立在两个一以贯之的设计主张之上。这两个主张决定了系统能否长期低成本演进。

### 1. 配置驱动：业务变化由配置应对，不依赖代码开发

资源对象、属性、校验规则、流程路由、外部接口，全部做成可配置：

- **资源对象与属性可配置**：每个资源对象的名称、标识、所属表、属性定义（是否必填、是否前端展示、是否上游提供、字段类型、维护方式、默认值等）均由配置管理维护。新增一类资源、新增一个属性，配置即可，无需开发。
- **校验规则可配置**：校验的本质统一抽象为"查询 + 比对 + 阈值判定"——通过 SQL 查库或 API 调外部系统取得数据，与工单数据比对，按设定阈值判定是否通过。运营团队配置规则即可，无需为每条规则写代码。
- **接口可配置**：与上下游系统的接口地址、认证、参数映射做成配置，上下游变动时调整配置即可，避免硬编码导致的连锁改造。

**这一理念的价值**：业务规则的维护权交还给最懂业务的运营团队与专业主管，系统对一线需求的响应从"提需求等厂家排期"变为"配置即时生效"，从根本上摆脱对外包研发力量的依赖。

### 2. 工单统一建模：一切业务场景皆为工单

系统把所有业务场景——入网、割接、搬迁、退网、暂时关停、租赁费等——统一建模为"工单"，通过"专业 + 场景"两个维度区分业务类型，复用同一套工单、流程、审批、资源数据体系，**不为每个新场景另建一套平行的业务系统**。

**这一理念的价值**：

- **扩展成本低**：新增一个业务场景，是在统一模型上做配置与少量场景特有逻辑，而非新建一个独立系统。
- **认知一致**：无论哪个专业、哪种场景，管理人员面对的都是"工单"这一致的心智模型，降低使用与维护门槛。

> **诚实边界**：系统研发启动于 2026 年 4 月，"通用骨架对所有场景复用、新场景只需轻量适配"是已被光缆入网验证的设计主张与演进方向，而非对全部现网场景的承诺。
```
