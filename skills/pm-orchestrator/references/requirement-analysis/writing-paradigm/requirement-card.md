# 需求卡片字段范式

> 前置读取：本文件依赖 `writing-paradigm/general-rules.md` 中的六条通用规律和六种语言范式定义。生成需求卡片草稿前，先读 `general-rules.md`，再读本文件。
>
> **素材来源**：撰写每个字段的最终润色值时，必须从字段 JSON 的 `qa_log` 中读取该字段的全部 Q&A 记录作为素材。`qa_log` 里的用户回答是原始事实来源，最终润色值是基于这些事实按范式写出的产物。只能比 Q&A 内容多（补充因果、结构化），不能比 Q&A 内容少（丢信息）。

---

## 字段范式速查

| 字段 | 范式 |
| --- | --- |
| 需求基本信息 | C-表头字段 |
| 现状描述 | D |
| 痛点 | A |
| 问题本质还原 | F |
| 需求评估结果 | C |

## 各字段详解与好差对比

### 需求基本信息

**范式**：C-表头字段（表格，每个 value 用"定性 + 具名细节"）

**规则**：表格中每个字段的值不能只写一个标签，要写"一句话定性 + 具名细节"。

**差写法**：

```
| 需求来源 | 双向坐实 |
| 提出人/角色 | 集团副总、一线人员 |
| 触发时间/时机 | 近期事项 |
| 影响范围 | 四个专业 |
| 当前状态 | 尚无系统支持 |
```

**好写法**：

```
| 需求来源 | 双向坐实--自上而下（集团副总战略要求、中心年度网络治理工作方案）与自下而上（广东省分公司管线人员迫切诉求）两股驱动力方向一致，同时存在 |
| 提出人/角色 | 集团副总经理（自上而下推动全流程贯通）、中心网络治理工作组（制定年度方案）、广东省分公司管线专业人员（自下而上迫切诉求） |
| 触发时间/时机 | 2026 年中心网络治理工作方案落地周期，集团要求四专业系统贯通与数据贯通 |
| 影响范围 | 光缆、传输、动环、无线四个专业，涉及建设方/外包厂家、维护方/运维团队、一线管线专业人员 |
| 当前状态 | 现有资源全生命周期系统由外包厂家建设，存在组织与技术双重债务，无源头数据质量管控能力 |
```

### 现状描述

**范式**：D（开头定位 + 流程图 + 关键特征）

**规则**：
1. 开头一句话定位当前业务在整个链条中的位置
2. 用 mermaid 或 text 流程图画出当前流程的关键步骤
3. 用"关键问题"或"关键特征"分条补充图中无法表达的重点

**差写法**：

```
当前流程：智能规建推送交维数据 -> 系统生成入网交维工单 -> 按专业/场景/网络层级路由到对应审批流程 -> 多级审批 -> 配置化数据校验（查询+比对+阈值判定）-> 校验通过则归档并推送资源中心，校验不通过则退回补充/修正后重新校验
```

> 差在哪：没有开头定位句；一行箭头流程不是图，只是逗号拼接的变体；没有关键特征分条。

**好写法**：

```
当前网络资源的入网交维管理散落在多个系统和线下工具中，缺乏统一的源头管控：

```text
智能规建系统（建设竣工）
    ↓ 钉钉/邮件通知交维
建设方手工录入资源数据到资源管理系统
    ↓ 纸质签字审批
维护方人工逐条核对资源数据
    ↓ 核对通过则交维完成
资源数据进入网络资源中心（事后可能不准）
```

关键问题：

- **交维靠线下、靠人情**：交维流程以钉钉、邮件、纸质签字为主，资源数据由建设方手工录入，"录得全不全、对不对"一线想知道但缺手段。
- **人工核对不可持续**：一条光缆系统包含大量资源对象，靠人工逐条核对既不现实也无精力，质量把关在人情关系下容易流于形式。
- **前清后乱**：集团与各省每一两年组织大规模资源数据清查，清查完很快又乱。现有的数据治理平台只能事后清理，无法在数据产生的源头管控。
```

### 痛点

**范式**：A（总结开头 + 分条列点）

**规则**：
1. 第一句总结痛点的性质（如"存在长期、具体的困难"）
2. 每条以 `**加粗关键词**` 开头，冒号后跟具体行为 + 频次/规模 + 损失后果
3. 至少 2 条，每条独立成段

**差写法**：

```
1. 人工核对不可持续：一条光缆系统包含大量资源对象，靠人工逐条核对既不现实也无精力，质量把关在人情关系下容易流于形式。2. 前清后乱顽疾：集团与各省每1-2年组织大规模资源数据清查，清查完很快又乱。现有数据治理平台只能事后清理，无法在数据产生的源头管控
```

> 差在哪：没有总结开头；编号后没有加粗关键词；两条压成一段；缺少损失后果。

**好写法**：

```
当前网络资源的入网交维管理存在长期、具体的困难：

- **资源是"哑资源"**：光缆等资源缺乏与网管采集数据的有效比对手段，资源数据是否准确、完整无从校验。导致政企业务开通核查时可能误判"资源不具备"，重复发起建设投资。
- **交维靠线下、靠人情**：交维流程以钉钉、邮件、纸质签字为主，资源数据由建设方手工录入资源管理系统，"录得全不全、对不对、和设计图纸是否一致"一线想知道但缺手段。
- **人工核对是灾难**：一条光缆系统包含大量资源对象，靠人工逐条核对既不现实也无精力，质量把关在人情关系下容易流于形式。
- **"前清后乱"顽疾**：集团与各省每一两年都会组织大规模资源数据清查，清查完很快又乱。现有的数据治理平台只能事后清理，无法在数据产生的源头管控。
- **流程与生产"两张皮"**：一线的生产作业已通过钉钉、邮件或本地自建审批完成，但这些流程与资源数据更新并不联动。若系统要求一线干完活后再到系统里重新走一遍流程，对一线就是与实际生产脱节的额外负担，结果多沦为事后补单。
```

### 问题本质还原

**范式**：F（段落论证）

**规则**：
1. 第一段摆事实：当前的具体现象和证据，不含结论
2. 第二段引出结论：用"这说明"、"因此"过渡，回答"本质是什么"
3. 可选 blockquote 标注更深层的意义或边界

**差写法**：

```
源头失控：数据质量管控不在数据产生的第一时间介入，而是事后补救。资源数据在录入和交维环节缺乏有效的质量门禁，导致脏数据流入系统后只能靠周期性清查治理，治理完又很快被新数据污染，形成前清后乱的恶性循环
```

> 差在哪：只有结论没有事实铺垫；一段压缩句，没有推理过程；没有"这说明/因此"的过渡。

**好写法**：

```
当前资源数据的录入和交维环节缺乏有效的质量门禁。交维流程以钉钉、邮件、纸质签字为主，资源数据由建设方手工录入，"录得全不全、对不对"一线想知道但缺手段。集团与各省每一两年组织大规模清查，但清查完很快又乱--现有数据治理平台只能在数据已经进入资源中心之后做事后清理，无法在数据产生的源头管控。

这说明问题的本质是**源头失控**：数据质量管控不在数据产生的第一时间介入，而是事后补救。脏数据流入系统后只能靠周期性清查治理，治理完又很快被新数据污染，形成"前清后乱"的恶性循环。要打破这个循环，必须在数据进入资源中心之前设立不可绕过的质量关卡。

> 更深一层：这不只是数据质量问题，而是管控方式问题。过去的思路是"先污染后治理"，用清查补漏；正确的方式是"源头把关"，让不合格数据在产生时就进不了系统。
```

### 需求评估结果

**范式**：C（表格，理由列要丰富）

**规则**：
1. 表格上方一句话说明评估的维度和逻辑
2. 理由列不能只写一句话，要有"结论 + 因为…"
3. 资源维度如果待验证，写明"待验证"和待验证的原因

**差写法**：

```
| 维度 | 评分/结论 | 理由 |
| --- | --- | --- |
| 业务价值 | 10 分 | 数据质量从事后清查前移到源头管控 |
| 影响 | 7 分 | 防重复投资 |
| 可行性 | 8 分 | 流程规范化 |
| 资源 | [待验证] | [待验证] |
```

> 差在哪：理由列太薄，只有半句话；没有因果链；资源维度没有写待验证的原因。

**好写法**：

```
从业务价值、影响范围、可行性和资源投入四个维度评估：

| 维度 | 评分/结论 | 理由 |
| --- | --- | --- |
| 业务价值 | 10 分 | 直接改变网络资源数据从产生到入库的管控方式。数据质量从事后清查前移到源头管控，校验通过率和关键字段完整率首次可统计、可考核。 |
| 影响 | 8 分 | 覆盖光缆、传输、动环、无线四个专业，涉及建设方、维护方、一线管线人员、各级管理部门。不做则生产管理仍然散乱，集团网络治理工作要求无法落地。 |
| 可行性 | 7 分 | 通用能力层已完成开发，光缆入网交维已端到端验证。主要不确定性在于上游接口字段对齐和多省数据查询的稳定性。 |
| 资源 | 待验证 | 当前核心研发集中度较高，需通过委外引入第三方力量补充。委外模式和管理方式待立项后细化确认。 |
```
