# Feature 字段范式

> 前置读取：本文件依赖 `writing-paradigm/general-rules.md` 中的六条通用规律和六种语言范式定义。生成 Feature 草稿前，先读 `general-rules.md`，再读本文件。
>
> **素材来源**：撰写每个字段的最终润色值时，必须从字段 JSON 的 `qa_log` 中读取该字段的全部 Q&A 记录作为素材。`qa_log` 里的用户回答是原始事实来源，最终润色值是基于这些事实按范式写出的产物。只能比 Q&A 内容多（补充因果、结构化），不能比 Q&A 内容少（丢信息）。

---

## 字段范式速查

| 字段 | 范式 |
| --- | --- |
| 需求背景 | 自动填入（引用需求卡片，无需追问） |
| 能力名称 | 简单字段（业务语言命名，能对应真实用户任务） |
| 能力描述 | F |
| 能力目标 | A |
| 用户角色 | 自动填入（引用 Epic 角色，说明本 Feature 由谁使用） |
| 业务价值 | A |
| 业务场景 | A |
| 业务流程 | D |
| 业务规则 | C |
| 技术可行性 | F |
| 资源投入 | C |
| 优先级 | 简单字段（P0/P1/P2 + 排序依据） |

## 各字段详解与好差对比

### 需求背景

**写法**：引用需求卡片，自动填入。格式为"本 Feature 回应 [[req-xxx]] 中的需求：{{相关背景}}"。无需追问，不套范式。

---

### 能力名称

**写法**：业务语言命名，能对应真实用户任务。非菜单名、模块名或技术名。一句话。

- **差**：`设备查询模块`
- **好**：`设备名称快捷查询`

---

### 能力描述

**范式**：F（段落，从用户视角描述前后状态变化）

**规则**：
1. 第一段描述用户使用该能力前的状态和困难
2. 第二段描述使用后能完成什么、状态变成什么
3. 从用户视角写，不从系统功能视角写

**差写法**：

```
用户输入设备名称或别名后，系统匹配候选设备，展示端口数量、端口占用、连接关系、数据来源和更新时间。
```

> 差在哪：只有系统功能描述，没有用户前后状态变化；没有"之前怎样"的铺垫。

**好写法**：

```
一线资源管理员在故障排查时，只知道设备名称，无法快速查到端口数量、端口占用和设备间连接关系，需要跨多个系统和 Excel 台账人工拼接，平均耗时 30-60 分钟。

使用本能力后，用户输入设备名称或别名，系统在 1 分钟内返回候选设备的端口数量、端口占用、连接关系，并标注每条数据的来源系统和更新时间。用户无需跨系统查询和手工拼接，直接在统一入口获取可信结果。
```

---

### 能力目标

**范式**：A（总结 + 分条，含量化指标）

**规则**：
1. 第一句总结能力目标的方向
2. 每条以加粗关键词开头，含"从X到Y"的量化变化
3. 至少 1 条可量化的指标

**差写法**：

```
让核心设备查询从 30 分钟降至 1 分钟以内，且每条结果都有来源标注。
```

> 差在哪：只有一句话，没有分条；没有总结开头。

**好写法**：

```
本能力要实现以下可验证的目标：

- **查询效率**：核心设备查询从 30-60 分钟降至 1 分钟以内。
- **结果可信度**：每条查询结果标注来源系统和更新时间，来源标注覆盖率 100%。
- **减少二次修正**：查询结果被人工二次修正比例低于 10%。
```

---

### 用户角色

**写法**：引用 Epic 角色，自动填入。需说明本 Feature 由哪个角色使用、谁只查看、谁审批。一句话或简短列表。

- **差**：`引用 Epic 中的角色`
- **好**：`引用 [[epic-001]] 中的角色：一线资源管理员（主要使用者）、网络运维工程师（查看结果）、专业主管（审批异常待办）`

---

### 业务价值

**范式**：A（总结开头 + 分条列点）

**规则**：
1. 第一句总结价值方向
2. 每条以加粗关键词开头，含对用户和业务的具体价值
3. 至少 1 条说明"不做会损失什么"

**差写法**：

```
减少跨系统查询和 Excel 拼接，降低错误判断风险。
```

> 差在哪：只有半句话；没有分条；没有"不做会损失什么"。

**好写法**：

```
本能力对用户和业务的价值：

- **减少查询耗时**：一线资源管理员不再需要跨多个系统和 Excel 台账人工拼接，单次查询节省 30-50 分钟。高峰期每天查询 10 次以上，累计节省显著。
- **降低错误判断风险**：查询结果带来源标注，用户可判断数据可信度，避免基于过时或不一致数据做错误判断。
- **为后续治理沉淀数据链路**：查询日志和异常数据登记为后续资源数据治理提供可信的数据基础。不做则数据口径不一致问题持续存在，故障定位时效无法改善。
```

---

### 业务场景

**范式**：A（总结 + 分条，每条含触发条件和用户目标）

**规则**：
1. 第一句总结场景覆盖范围
2. 每条以加粗场景名开头，含"什么时候触发 + 用户要完成什么"
3. 至少 1 个场景

**差写法**：

```
1. 故障排查时快速确认设备端口占用。2. 割接前核实设备间连接关系。3. 发现数据异常时提交资源待办。
```

> 差在哪：没有总结开头；编号后没有加粗关键词；缺少"用户从什么状态到什么状态"的描述。

**好写法**：

```
本能力在以下业务场景中触发：

- **故障排查**：运维工程师收到故障告警后，需要快速确认故障设备的端口占用和上联关系，以判断故障影响范围和定位根因。从"跨系统查 30 分钟"到"1 分钟内查到"。
- **割接前核实**：割接方案制定前，工程师需要核实涉及设备间的连接关系，确认割接影响范围。从"靠 Excel 台账核对"到"系统直接展示"。
- **数据异常登记**：查询过程中发现数据异常时，用户直接提交资源待办，从"发现问题但无人跟进"到"异常进入待办闭环"。
```

### 业务流程

**范式**：D（开头定位 + 流程图 + 关键步骤）

**规则**：
1. 开头一句话定位该流程的触发条件和最终结果
2. 用 mermaid 或 text 流程图画出关键步骤
3. 用分条补充异常分支和兜底策略

**差写法**：

```
输入设备名称 -> 匹配候选设备 -> 展示端口与连接关系 -> 标注来源 -> 异常提交待办
```

> 差在哪：只有一行箭头流程；没有开头定位；没有异常分支；没有加粗关键词。

**好写法**：

```
用户从输入设备名称到获取查询结果，中间经过匹配、展示和异常处理：

```text
用户输入设备名称/别名
    -> 系统匹配候选设备（精确匹配优先，别名和模糊匹配作为候选）
    -> 展示端口数量、端口占用、连接关系
    -> 每条结果标注来源系统和更新时间
    -> 用户判断数据是否可信
        -> 数据正常：查询完成
        -> 数据异常：提交资源待办，进入待验证清单
```

关键步骤说明：

- **名称匹配**：精确匹配优先。如果有多个候选设备，展示候选列表让用户选择。如果无匹配结果，提示用户检查名称或尝试别名。
- **来源标注**：每条端口和连接关系必须显示来源系统和更新时间。如果某字段无来源数据，显示"数据缺失"而非留空。
- **异常隔离**：数据冲突时不得覆盖主数据，只进入待验证清单。用户可标注冲突原因，流转给数据责任人处理。
```

### 业务规则

**范式**：C（表格，含异常处理列）

**规则**：
1. 表格至少 3 列：规则项、规则说明、异常/边界处理
2. 异常处理列不能为空，要写"如果规则被违反，系统怎么处理"

**差写法**：

```
名称匹配：精确匹配优先，别名和模糊匹配作为候选。来源标注：每条数据必须显示来源。异常隔离：不覆盖主数据。
```

> 差在哪：没有表格；没有异常处理列；压缩成一段。

**好写法**：

```
| 规则项 | 规则说明 | 异常/边界处理 |
| --- | --- | --- |
| 名称匹配 | 精确匹配优先，别名和模糊匹配作为候选 | 无匹配结果时提示用户检查名称或尝试别名；多个候选时展示列表让用户选择 |
| 来源标注 | 每条端口和连接关系必须显示来源系统和更新时间 | 某字段无来源数据时显示"数据缺失"而非留空，并计入数据完整率统计 |
| 异常隔离 | 数据冲突时不得覆盖主数据，只进入待验证清单 | 用户可标注冲突原因，待办流转给数据责任人处理；冲突数据仍可查看但不作为可信来源 |
| 并发查询 | 多用户同时查询同一设备不互相阻塞 | 查询为只读操作，无并发冲突；待办提交时按时间戳排序 |
```

### 技术可行性

**范式**：F（段落论证 + 依赖清单分条）

**规则**：
1. 第一段评估整体可行性，给出结论
2. 第二段列出关键依赖和最大不确定性
3. 用分条列出依赖清单，每条注明"是否已确认"

**差写法**：

```
需接入设备台账、端口占用和连接关系数据；若连接关系口径不一致，先做来源标注和人工核实，不自动合并。
```

> 差在哪：只有半句话；没有可行性结论；依赖没有分条；没有标注是否已确认。

**好写法**：

```
整体可行性评估：中高。核心数据（设备台账、端口占用）在现有系统中可读取，主要不确定性在于连接关系数据的口径一致性和更新频率。

关键依赖：

- **设备台账数据**：已确认可用，来源为资源管理系统，按日更新。
- **端口占用数据**：已确认可用，来源为网管采集系统，准实时更新。
- **连接关系数据**：待验证。不同专业的连接关系口径可能不一致，需先验证数据完整性。若口径不一致，先做来源标注和人工核实，不自动合并。
- **用户权限系统**：已确认可用，复用统一门户 SSO。

> 最大风险：连接关系数据质量。如果连接关系数据不完整或不一致，首版先支持来源标注和人工核实，不自动合并，避免给用户错误结果。
```

### 资源投入

**范式**：C（表格，含角色/规模/外部依赖）

**规则**：
1. 表格至少 3 列：角色/资源、投入规模、备注/外部依赖
2. 投入规模要量化（人天/人周）
3. 如果资源不足，用 blockquote 写明"先做哪部分、暂缓哪部分"

**差写法**：

```
预计 2 名后端、1 名前端、1 名测试投入 4-6 周；需资源数据接口人配合确认口径。
```

> 差在哪：没有表格；没有分角色列出；外部依赖没有分条。

**好写法**：

```
| 角色/资源 | 投入规模 | 备注/外部依赖 |
| --- | --- | --- |
| 后端开发 | 2 人 × 4-6 周 | 负责数据接入、查询逻辑和来源标注 |
| 前端开发 | 1 人 × 3-4 周 | 负责查询界面和结果展示 |
| 测试 | 1 人 × 2-3 周 | 负责功能测试和数据质量验证 |
| 资源数据接口人 | 按需配合 | 确认设备台账、端口占用和连接关系的数据口径 |

> 如果资源只能做一半：先做设备名称查询和端口占用展示（P0），连接关系展示和异常待办放 P1。核心查询能力先上线，依赖数据质量的能力后续补齐。
```

---

### 优先级

**写法**：`P0/P1/P2` + 排序依据。排序依据要写"因为…，不先做会阻塞…"。一句话。

- **差**：`P0`
- **好**：`P0（排序依据：高频场景、直接减少查询耗时，是后续连接关系展示和异常待办的前提）`
