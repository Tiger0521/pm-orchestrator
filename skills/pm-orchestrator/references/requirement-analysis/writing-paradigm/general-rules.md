# 需求分析写作范式 · 总则

本文件定义所有字段共通的写作规律和语言范式。生成任何文档草稿前必须先读取本文件，再读取对应文档的字段范式文件。

---

## 六条通用规律

不论用哪种范式，所有字段的文字都必须遵守以下六条。任一条不满足，视为字段质量不达标。

### 1. 总结先行

每个字段的第一句话是一句总结性判断，概括这个字段要说的核心意思。不要上来就直接罗列条目。

- **差**：`1. 人工核对不可持续：... 2. 前清后乱顽疾：...`
- **好**：`当前网络资源的入网交维管理存在长期、具体的困难：` 然后分条

### 2. 加粗关键词领条

分条列点时，每条以 `**加粗关键词**` 开头，后接冒号或句号和详细解释。关键词应是有信息量的判断或意象，不是泛词。

- **差关键词**：`问题一`、`要点1`、`人工核对`（太泛）
- **好关键词**：`人工核对是灾难`、`"前清后乱"顽疾`、`流程与生产"两张皮"`（有判断、有意象）

### 3. 具名细节

解释里使用具体的部门名、工具名、人名、频次、数字，不用抽象词。能量化就量化，不能量化就举一个具名例子。

- **差**：`交维流程以线下方式为主，效率低`
- **好**：`交维流程以钉钉、邮件、纸质签字为主，资源数据由建设方手工录入资源管理系统`
- **差**：`定期组织清查`
- **好**：`每一两年都会组织大规模资源数据清查`

### 4. 结论跟"为什么"

每个结论后面跟理由或因果链。用"不是…而是…"、"因为…所以…"、"…的原因是…"等句式把推理过程写出来。读者看到结论后脑中冒出的第一个问题应该是"为什么"，而"为什么"的答案就在下一句。

- **差**：`选择与现有系统完全隔离`
- **好**：`选择与现有系统**完全隔离**。现有系统由外包厂家分前后端建设，存在严重的组织与技术双重债务：厂家持续裁员、能力下滑……在 capex 投资收紧的背景下，继续依赖厂家建设难以协调推进。`

### 5. 诚实标注

区分事实、推断和待验证内容。用 blockquote（`>`）标注边界声明、关键假设、责任划分。无法确认的内容写"待验证"，不假装确定。

- **好**：`> **诚实边界**：系统研发启动于 2026 年 4 月，"通用骨架对所有场景复用"是已被光缆入网验证的设计主张，而非对全部现网场景的承诺。`

### 6. 视觉结构匹配内容

内容有多条并列要点时用列表；有结构化对比时用表格；有流程或流转时用 mermaid 或 text 流程图。不要把所有信息压成一段流水句。

| 内容形态 | 视觉结构 |
| --- | --- |
| 多条并列要点（痛点、价值、目标） | 加粗关键词领条的列表 |
| 角色对比、规则对比、评估对比 | 表格（含关联列，不能只有名称+描述） |
| 流程、流转、状态转换 | mermaid flowchart 或 text 流程图 |
| 分层、分档、分优先级 | 小标题 + 每层表格或分条 |
| 深度论证（从事实到结论的推理） | 两段递进段落 |

## 六种语言范式

### 范式 A · 总结开头 + 分条列点

最基础、最高频的范式。适用于"有多个并列要点需要展开"的字段。

**结构**：

```
一句话总结性判断（概括这些要点的共性）：

- **关键词1**：详细解释，含具名细节和因果
- **关键词2**：详细解释，含具名细节和因果
- **关键词3**：详细解释，含具名细节和因果
```

**规则**：
1. 第一句是总结判断，不是"以下有几个要点"这类空话
2. 至少 2 条，每条独立成段
3. 每条以 `**加粗关键词**` 开头，冒号后跟解释
4. 解释里至少包含一个具名细节（人/工具/频次/数字）

---

### 范式 B · blockquote 核心论断 + 分要点展开

适用于"需要先给出一个核心论断，再拆解其内涵"的字段。

**结构**：

```
> **一句话核心论断（加粗，放在 blockquote 中）**

这句话有 N 个要点：

1. **要点关键词**。解释段落，含因果和具名细节。
2. **要点关键词**。解释段落，含因果和具名细节。
3. **要点关键词**。解释段落，含因果和具名细节。
```

**规则**：
1. 核心论断用 blockquote + 加粗，是整个字段的"题眼"
2. 论断后用"这句话有 N 个要点"过渡到分条
3. 要点用编号列表（`1.` `2.` `3.`），每条以加粗关键词开头，后接句号和解释段落
4. 解释段落要比范式 A 更充分，每个要点是一个完整的论证段落

---

### 范式 C · 开头定位 + 表格

适用于"有多个同构对象需要对比"的字段。

**结构**：

```
一句话说明分类维度和逻辑：

| 列1 | 列2 | 列3（关联列） |
| --- | --- | --- |
| 值 | 值 | 值 |
```

**规则**：
1. 表格上方必须有一句话说明"按什么维度分类"或"覆盖什么范围"
2. 表格至少 3 列，其中最后一列应为"关联列"--对应理由、痛点、异常处理等，不能只有名称+描述两列
3. 单元格内容不能只有标签，要有一句话解释
4. 表格下方可跟 blockquote 补充说明或边界声明

**变体 C-表头字段**：需求基本信息也是表格形式（key-value），每个 value 应遵循"定性 + 具名细节"的写法，不能只写标签。

---

### 范式 D · 开头定位 + 流程图 + 关键特征分条

适用于"描述流程、现状或场景"的字段。

**结构**：

```
一句话定位该流程/现状/场景在整个业务中的位置：

```mermaid
flowchart ...
```

关键特征：

- **特征关键词1**：解释
- **特征关键词2**：解释
```

**规则**：
1. 开头一句话定位--这个流程在整体业务中处于什么位置，上承什么、下接什么
2. 流程图用 mermaid `flowchart` 或 text 箭头图，节点用业务语言命名
3. 流程图后用"关键特征"或"关键步骤"分条补充图中无法表达的重点
4. 如果流程有异常分支，图中用条件标注（`-->|校验不通过|`）

---

### 范式 E · 分层结构 + 每层表格/分条

适用于"需要分层、分档展示"的字段。

**结构**：

```
一句话说明分层逻辑：

**第一层 · 层名**

（表格或分条）

**第二层 · 层名**

（表格或分条）

**第三层 · 层名**

（表格或分条）
```

**规则**：
1. 开头一句话说明"按什么维度分层"
2. 每层用 `**层名**` 小标题领起
3. 每层内用表格或分条，内容结构一致
4. 层与层之间有递进或优先级关系，不是平级罗列

---

### 范式 F · 段落论证

适用于"需要从事实推导出结论"的字段。

**结构**：

```
第一段：摆事实（现状、证据、具名细节，不含结论）

第二段：引出结论（"这说明…"、"因此…"、"这意味着…"）

> （可选）blockquote 标注更深层的意义或边界
```

**规则**：
1. 第一段只摆事实和证据，不下结论
2. 第二段用过渡词（"这说明"、"因此"、"这意味着"）引出结论
3. 结论要回答"本质是什么"或"因此该怎么做"
4. 可选 blockquote 标注更深层的意义、边界或假设

## JSON 编码注意事项

字段 JSON 中的值是字符串，多行 markdown 内容（列表、表格、mermaid 代码块）需要在 JSON 中用 `\n` 表示换行。`render-doc.sh` 会将 `\n` 转换为实际换行符后输出到 Markdown 文件。

**规则**：
1. JSON 字符串中的换行用 `\n`，段落间距用 `\n\n`
2. JSON 字符串中的双引号用 `\"` 转义
3. JSON 字符串中的字面反斜杠用 `\\` 转义（`render-doc.sh` 优先解码 `\\` → `\`，再解码 `\n`/`\t`/`\"`）
4. 表格中的 `|` 字符在 JSON 中不需要转义
5. 草稿预览（对话中展示）与字段 JSON 和最终渲染的 Markdown 三者必须同结构、同字段、同正文内容
