# 需求分析问题库

## 使用原则

- 每轮只问一个字段。一个字段有多个信息点时，用一个综合问题覆盖。
- 不机械问满；已有答案不重复问，信息缺口决定下一问。
- 先问证据和现状，再问判断和评估。
- 听到”平台””AI 赋能””竞品都有””领导要求””都要做”时，先做前提挑战。
- 回答不清时追问 1-2 轮，仍说不清标记为”待验证”并继续。
- 建设思路、范围边界、业务流程、业务规则、资源投入、优先级必须直接问用户，不靠 AI 推导。
- 问题只用于澄清信息，不规定写作格式。

---

## 问题自检

每个问题在提出前自检三件事：

| 元素 | 使用方式 |
| --- | --- |
| 覆盖内容 | 这一问要覆盖哪个字段的哪些信息点：具体角色、真实场景、当前行为、证据数据、业务结果、优先级排序。 |
| 红灯信号 | 判断回答是否逃避：只讲口号、只讲功能、只讲权威来源、只说”都要”、没有证据或边界。 |
| 追问兜底 | 如果回答不清，追问什么：具体事实、频次规模、前后步骤、依赖关系。 |

不要把自检内容输出给用户；它用于选择问题、判断回答质量、决定是否追问。

## 字段覆盖确认问题

本节用于阶段性回收信息，防止追问偏离最终产物。回执只做归纳，不替用户下结论；回执后只问一个确认或补充问题。

字段确认必须采用逐字段清单格式：每个字段都要展示“字段名 / 完整内容 / 状态（已确认、待验证、缺失）”。禁止只写字段名、信息组名称或“全部字段已覆盖”。用户必须看到具体内容后才能确认。

### 需求卡片字段确认

在输出需求卡片草稿前，把已问到的内容回收给用户确认：

- 典型问题：我先按字段回收需求卡片目前可写的信息，请确认具体内容是否准确：
  - 基本信息：{{basic_info}}；状态：{{basic_info_status}}
  - 现状描述：{{current_state}}；状态：{{current_state_status}}
  - 痛点：{{pain_points}}；状态：{{pain_points_status}}
  - 问题本质：{{root_problem}}；状态：{{root_problem_status}}
  - 需求评估结果：业务价值 {{business_value_evaluation}}；影响 {{impact_evaluation}}；可行性 {{feasibility_evaluation}}；资源 {{resource_evaluation}}；状态：{{evaluation_status}}
  以上内容是否准确，哪里需要补充或纠正？
- 红灯信号：用户只确认了部分字段，但其他字段仍为缺失且未标记为待验证。

### Epic 字段确认

在输出 Epic 草稿前，把已问到的内容回收给用户确认：

- 典型问题：我先按字段回收 Epic 目前可写的信息，请确认具体内容是否准确：
  - 产品名称：{{product_name}}；状态：{{product_name_status}}
  - 定位：{{positioning}}；状态：{{positioning_status}}
  - 目标：{{goals}}；状态：{{goals_status}}
  - 用户角色：{{roles}}；状态：{{roles_status}}
  - 核心场景：{{scenarios}}；状态：{{scenarios_status}}
  - 产品价值：{{value}}；状态：{{value_status}}
  - 范围与边界：{{scope}}；状态：{{scope_status}}
  - 建设思路：{{build_approach}}；状态：{{build_approach_status}}
  以上内容是否准确，哪里需要补充或纠正？
- 红灯信号：有产品功能清单，但没有产品定位、目标、范围边界或建设思路。

### Feature 字段确认

在输出每个 Feature 草稿前，把已问到的内容回收给用户确认：

- 典型问题：针对 {{feature_name}}，我先按字段回收 Feature 目前可写的信息，请确认具体内容是否准确：
  - 能力名称：{{feature_name}}；状态：{{feature_name_status}}
  - 能力描述：{{description}}；状态：{{description_status}}
  - 目标：{{goal}}；状态：{{goal_status}}
  - 用户角色：{{roles}}；状态：{{roles_status}}
  - 业务价值：{{value}}；状态：{{value_status}}
  - 场景：{{scenario}}；状态：{{scenario_status}}
  - 流程：{{process}}；状态：{{process_status}}
  - 规则：{{rules}}（含异常和边界处理）；状态：{{rules_status}}
  - 可行性：{{feasibility}}；状态：{{feasibility_status}}
  - 资源投入：{{resources}}；状态：{{resources_status}}
  - 优先级：{{priority}}；状态：{{priority_status}}
  以上内容是否准确，哪里需要补充或纠正？
- 红灯信号：只得到能力名称，没有业务流程、规则、资源投入或优先级。

## 广度优先问题库

用户初始介绍覆盖多个角色、多个流程、多个痛点或一个”大系统/大平台”时，先用本节从广度了解全貌。不要一上来就追问某个细节。

### 当前理解回执

每次进入下一问前，用 2-5 行给用户反馈当前理解：

```text
我先把你的描述摊开看：
- 目前我理解的用户角色：{{roles}}
- 主要场景/环节：{{scenarios}}
- 可能的问题簇：{{problem_clusters}}
- 候选能力方向：{{capability_candidates}}
- 还不确定：{{unknowns}}
```

然后只问一个问题，例如：”以上理解有没有漏掉或放错重点的地方？”

### 1. 角色

- 追问目标：找全直接使用者、协作者、审批/决策者、受影响者。
- 红灯信号：只盯住第一个被提到的角色；把部门、团队、用户群当成具体角色。
- 设计理由：大需求常常跨角色，少一个角色就会把后续卡片写窄。
- 典型问题：我先列一下目前可能涉及的角色：{{roles}}。请确认是否还有遗漏，尤其是协作者、审批者或只接收结果的人？

### 2. 场景

- 追问目标：列出核心业务场景和发生环节，不急着深入单个场景。
- 红灯信号：只抓一个高频词往下问，忽略用户描述中的其他场景。
- 设计理由：场景梳理决定需求卡片和 Epic 是整体闭环，还是多个并列范围。
- 典型问题：这些问题分别发生在哪些业务环节？我目前理解有 {{scenarios}}，哪个漏了或放错了？

### 3. 问题簇

- 追问目标：把多个痛点分组，判断并列、因果、上下游或同一流程不同环节。
- 红灯信号：把多个真实问题强行压成一个”最重要问题”；或只沿着一个小痛点深挖。
- 设计理由：避免最后输出只覆盖大问题里的一个很小切片。
- 典型问题：我把问题暂时分成 {{problem_clusters}}。这些问题之间是什么关系：并列、因果、上下游，还是同一流程的不同环节？

### 4. 能力候选

- 追问目标：先列出可能的产品能力方向，再决定哪些进入 Epic、哪些拆成 Feature。
- 红灯信号：把用户说的第一个功能当成全部方案；直接进入页面/按钮/字段细节。
- 设计理由：能力候选能防止文档只写成某个功能点，而不是产品层解决方案。
- 典型问题：为了覆盖这些问题，可能需要哪些能力？我先列 {{capability_candidates}}，你看哪些必要、哪些暂缓、哪些不该做？

### 5. 范围确认

- 追问目标：确认本轮文档覆盖整体问题，还是先覆盖一个子问题。
- 红灯信号：AI 自行决定“先写一个小点”，但用户以为还在讨论整体需求。
- 设计理由：范围未确认时，输出很容易偏离用户的大问题。
- 典型问题：接下来输出文档时，是先覆盖整体闭环，还是先把 {{sub_problem}} 作为第一张需求卡片/第一个 Epic？如果只先做子问题，哪些内容需要暂缓？

## 需求卡片字段追问

需求卡片共 5 个必填字段。逐字段追问，每轮只问一个。回答不清时追问 1-2 轮，仍说不清标记为”待验证”。

### 需求基本信息

- 覆盖内容：需求来源、提出人/角色、触发时间/时机、影响范围、当前状态。
- 典型问题：这条需求是谁提出来的？什么时候触发的？影响哪些人、哪些业务？
- 回答不清时追问：是用户投诉、PM 规划、领导要求还是数据驱动？影响范围是某个团队还是全公司？
- 红灯信号：说不出提出人或触发原因；只说”业务需要”。

### 现状描述

- 覆盖内容：当前业务流程的关键步骤、现有替代方案、当前为什么做不到。
- 典型问题：现在这个事是怎么做的？从开始到结束经过哪几步？有没有用 Excel、群聊或其他工具在绕？
- 回答不清时追问：当前流程中哪一步最慢、最容易出错？绕过方案的成本是什么？
- 红灯信号：只说”效率低””系统分散”，说不出具体步骤和替代方案。

### 痛点

- 覆盖内容：当前最具体的返工、等待、错误、成本或风险。
- 典型问题：当前做法带来的最具体问题是什么？是返工、等待、错误、成本还是风险？有没有具体例子？
- 回答不清时追问：这个问题多久发生一次？影响多少人？每次损失多少时间或多少钱？
- 红灯信号：只说”体验不好””用户不满意”，没有具体行为和频次。

### 问题本质还原

- 覆盖内容：不看具体功能，这个问题本质上是哪类业务断点或管理缺口。
- 典型问题：如果不看具体功能或页面，这个问题的本质是什么？是信息不对称、流程断点、数据缺失还是决策滞后？
- 回答不清时追问：如果不用系统，这个问题在业务上意味着什么？它和上下游的关系是什么？
- 红灯信号：把问题本质写成功能需求；把”缺一个审批页面”当问题本质。

### 需求评估结果

- 覆盖内容：从业务价值、影响、可行性、资源四个维度评估，每个维度给出结论或评分，并说明理由。
- 典型问题：请从四个角度评估这条需求：1）业务价值：它能带来什么业务收益或管理价值？2）影响：不做会影响哪些对象、范围、频次、成本或风险？3）可行性：数据、流程、系统、技术和依赖是否具备？4）资源：需要哪些角色、投入量级和外部配合？每个角度请给出结论或评分以及理由。
- 回答不清时追问：业务价值能否落到指标或管理收益？影响对象和频次是什么？最大的可行性阻碍是什么？资源不足时先砍哪部分？
- 红灯信号：只给分不解释；只说”领导要求”；只评估价值和可行性，不评估影响范围或资源投入。

## Epic 字段追问

Epic 共 9 个必填字段。需求背景引用需求卡片，其余 8 个字段逐字段追问。

### 需求背景

- 本字段引用需求卡片，无需单独追问。
- 在 Epic 中写为”本 Epic 派生自 [@req-xxx]：{{需求卡片一句话结论}}”。

### 产品名称

- 覆盖内容：产品名称或工作名。
- 典型问题：这个产品（或产品能力）准备叫什么？如果暂未定名，先用什么工作名？
- 回答不清时追问：这个名称对外部或非项目成员是否可理解？
- 红灯信号：名称像项目代号或技术模块名，无法表达产品含义。

### 产品定位

- 覆盖内容：面向谁、在什么场景下、解决什么核心问题。
- 典型问题：一句话说，这个产品面向谁？在什么场景下？解决什么核心问题？
- 回答不清时追问：如果只留一句话给非项目成员介绍它，那句话是什么？
- 红灯信号：只说”做一个平台/系统/工具”，没有用户、场景和问题。

### 产品目标

- 覆盖内容：达成后希望改变的 1-3 个业务结果或指标。
- 典型问题：这个产品做成后，希望改变哪 1-3 个业务结果或指标？目标值是多少？
- 回答不清时追问：如果成功了，什么指标会变化？从多少变成多少？
- 红灯信号：只说”提升效率””优化体验”，没有具体指标和数值。

### 用户角色

- 覆盖内容：Epic 覆盖的用户角色，每个角色的核心诉求。
- 典型问题：这个产品覆盖哪些用户角色？每个角色的核心诉求分别是什么？
- 回答不清时追问：除了直接使用者，还有谁参与、审批、查看结果或受影响？
- 红灯信号：只说”业务部门””管理员”，没有具体角色和诉求。

### 核心场景

- 覆盖内容：产品覆盖的核心业务场景，至少 1 个。
- 典型问题：用户会在哪些具体业务场景下使用这个产品？哪个场景最常见、最痛或最优先？
- 回答不清时追问：用户从什么状态开始，经过什么动作，到达什么结果？
- 红灯信号：只说”日常使用””各种场景”，没有具体场景描述。

### 产品价值

- 覆盖内容：对用户、业务和战略的价值分别是什么。
- 典型问题：这个产品对用户、业务和战略的价值分别是什么？有没有不做会损失什么的判断？
- 回答不清时追问：用户愿意为它改变现有习惯吗？业务侧谁来买单或推动？
- 红灯信号：只说”很有价值””大家都需要”，没有区分用户价值和业务价值。

### 产品范围与边界

- 覆盖内容：哪些必须做、哪些明确不做、哪些后续再做或待验证。
- 典型问题：这个产品的范围内包含什么？明确不做什么？哪些放到后续版本或待验证？
- 回答不清时追问：如果有人问”为什么不包含 XX”，你的回答是什么？
- 红灯信号：只说”都要做”，没有范围外和暂缓事项。

### 建设思路

- 覆盖内容：系统建立在什么设计主张之上、每个主张的内涵和价值、这些主张如何决定系统能否长期低成本演进。
- 典型问题：这个系统不是功能的简单堆砌，它建立在什么设计主张之上？每个主张的内涵是什么？它带来什么价值？有没有哪些主张是方向而非承诺？
- 回答不清时追问：如果不用这个设计主张，会怎样？这个主张在哪些场景已验证，哪些还是方向？每个主张里具体什么东西可配置/可复用？
- 红灯信号：只列功能清单或优先级排序，没有设计理念；只说"配置化""平台化"但没有展开内涵和价值。

## Feature 字段追问

Feature 共 12 个必填字段。需求背景和用户角色引用已有文档，其余字段逐字段追问。每个 Feature 单独追问，不要用同一个问题覆盖多个 Feature。

### 能力清单（先问，确认后再逐个 Feature 深入）

- 覆盖内容：从 Epic 拆出的产品能力列表、每个能力解决哪类用户任务、能力边界初判。
- 典型问题：基于这个 Epic，需要拆成哪些产品能力？每个能力分别解决哪类用户任务？哪些能力必须先做，哪些暂缓或不做？
- 回答不清时追问：这些能力之间是否有依赖关系？是否都属于同一个产品闭环？
- 红灯信号：还没确认能力清单，就深挖某一个能力的细节；或能力像菜单名、模块名。

### 需求背景

- 本字段引用需求卡片，无需单独追问。
- 在 Feature 中写为”本 Feature 回应 [@req-xxx] 中的需求：{{相关背景}}”。

### 能力名称

- 覆盖内容：用业务语言命名的能力名称。
- 典型问题：这个能力用业务语言叫什么？如果对非技术人员说这个名字，他能理解是做什么的吗？
- 回答不清时追问：如果不用”管理””配置””查询”等泛词，换个更具体的说法？
- 红灯信号：能力名称像菜单名、模块名、技术名，不能对应真实用户任务。

### 能力描述

- 覆盖内容：这个能力让用户完成的具体任务。
- 典型问题：这个能力让用户完成哪件具体任务？从用户视角描述，不要从系统视角描述。
- 回答不清时追问：用户用这个能力之前是什么状态，用完之后是什么状态？
- 红灯信号：描述像功能说明书，看不出用户任务和业务结果。

### 能力目标

- 覆盖内容：这个能力完成后，哪个业务结果会改善。
- 典型问题：这个能力完成后，哪个业务结果会改善？用什么指标判断它做成了？
- 回答不清时追问：如果这个能力上线了但没人用，可能是哪里出了问题？
- 红灯信号：目标和 Epic 的产品目标无关，或没有可验证的标准。

### 用户角色

- 本字段引用 Epic 的用户角色，无需单独追问。
- 如果 Epic 有多个角色，需明确这个 Feature 由哪个角色使用、谁只查看、谁审批。

### 业务价值

- 覆盖内容：这个能力对用户和业务的具体价值。
- 典型问题：这个能力对用户和业务分别带来什么价值？不做会影响什么？
- 回答不清时追问：用户愿意为这个能力改变现有习惯吗？如果只做这一个能力，够不够解决一个最小闭环？
- 红灯信号：只说”提升效率””改善体验”，没有具体价值判断。

### 业务场景

- 覆盖内容：这个能力发生的具体业务场景，至少 1 个。
- 典型问题：这个能力在哪些具体业务场景下触发？用户从什么状态开始，到完成什么任务结束？
- 回答不清时追问：场景中用户的前后步骤是什么？这个能力在哪个环节介入？
- 红灯信号：场景描述像系统操作步骤，没有业务背景和用户动机。

### 业务流程

- 覆盖内容：从触发到结果的关键步骤、触发条件、操作角色、系统行为、输出结果。
- 典型问题：从这个能力被触发到得到结果，中间经过哪些关键步骤？每步由谁操作、系统做什么、输出什么？
- 回答不清时追问：如果某个步骤失败或超时，用户会看到什么？下一步怎么办？
- 红灯信号：只描述正常流程，没有异常、失败、超时、重复或权限不足的处理。

### 业务规则

- 覆盖内容：影响处理结果的条件、权限、校验、异常处理。
- 典型问题：哪些条件会影响处理结果？权限怎么控制？数据怎么校验？失败、重复、无权限时怎么处理？
- 回答不清时追问：如果同一份数据被两个人同时操作，应该怎么处理？
- 红灯信号：只讲正常流程规则；没有权限、冲突、异常和边界规则。

### 技术可行性

- 覆盖内容：依赖的系统、数据、接口，当前最大的技术不确定性。
- 典型问题：实现这个能力依赖哪些系统、数据、接口？当前最大的技术不确定性是什么？
- 回答不清时追问：有没有哪个依赖方还没沟通或确认？数据是否已经存在且可用？
- 红灯信号：只说”技术上没问题”，没有技术评估依据。

### 资源投入

- 覆盖内容：需要哪些角色、多少人天/人周，或哪些外部资源。
- 典型问题：实现这个能力大概需要哪些角色、多少人天或人周？有没有需要外部协调的资源？
- 回答不清时追问：如果资源只能做一半，先做哪部分？哪些可以暂缓？
- 红灯信号：只说”需要开发”但没有规模判断；或资源估算和建设思路矛盾。

### 优先级

- 覆盖内容：P0/P1/P2 及排序依据，不先做的阻塞关系。
- 典型问题：这个能力的优先级是什么（P0/P1/P2）？排序依据是什么？和其他能力相比为什么先做或后做？不先做会阻塞什么？
- 回答不清时追问：如果只能选一个 P0，是这个还是另一个能力？为什么？
- 红灯信号：只标 P0/P1/P2 标签，没有价值依据、依赖判断或阻塞关系；或所有能力都标 P0。

## 反谄媚禁用词表

这些表达会把需求分析变成附和。遇到模糊、方案先行或证据不足的回答时，禁止使用左列表达，改用右侧方式追问。

| 禁止用语 | 要求改写 |
| --- | --- |
| “这个想法很好/很有价值” | “这个想法成立需要先证明：谁在什么场景下真的需要它。” |
| “这个方向可以考虑” | “当前缺少的判断是：它解决哪个具体痛点，以及不做会损失什么。” |
| “这个功能确实重要” | “重要到什么程度？影响人数、频次、成本或风险是多少？” |
| “平台化方向比较完整” | “这些能力是否属于同一用户、流程、数据对象或管理闭环？” |
| “AI 很适合这个场景” | “AI 替代哪一步人工判断？数据是否足够？错误由谁兜底？” |
| “领导要求所以优先级高” | “领导要求背后的业务压力、验收口径和不做后果是什么？” |
| “竞品做了说明有需求” | “竞品用户和我们是否相同？我们自己的用户证据是什么？” |
| “体验可以继续优化” | “哪个具体行为变好了？用什么可观察标准判断体验改善？” |
| “后续可以逐步完善” | “第一步验证什么？哪些暂缓？哪些明确不做？” |
| “用户应该会喜欢” | “有没有用户行为、访谈、投诉、绕路成本或数据证据支撑？” |

规则：可以语气礼貌，但结论必须有立场；不能用“可以考虑、可能、建议关注”掩盖关键判断缺口。

## 前提挑战模式

| 用户表达 | 不接受的弱回应 | 应追问的方向 |
| --- | --- | --- |
| “做个平台/系统” | “平台方向很好” | 它要解决哪些具体问题？这些问题是否属于同一用户、流程、数据对象或管理闭环？ |
| “都要做” | “可以全量规划” | 哪些必须现在做？哪些暂缓？哪些明确不做？为什么？ |
| “提升效率” | “效率提升很重要” | 当前耗时是多少？目标耗时是多少？影响人数、频次和损失是什么？ |
| “智能化/AI 赋能” | “AI 很适合这个场景” | AI 替代哪一步判断？数据是否具备？错了谁兜底？不用 AI 能否解决？ |
| “按领导要求” | “领导要求优先级高” | 领导要解决的业务压力是什么？验收口径是什么？不做后果是什么？ |
| “竞品都有” | “竞品做了说明有需求” | 竞品用户和我们相同吗？我们自己的用户证据是什么？差异化价值是什么？ |
| “体验要好” | “可以优化体验” | 哪个具体场景、哪个具体行为、哪个可验证标准代表体验变好？ |

## 复杂度路由

| 情况 | 提问深度 |
| --- | --- |
| 小范围迭代 | 只追问增量背景、受影响角色、规则边界、验收口径和上线风险。 |
| 新产品/新闭环 | 完整追问需求卡片和 Epic 全部字段，再进入 Feature 能力清单。 |
| 高风险/强依赖 | 追加追问数据来源、上下游依赖、权限合规、失败兜底和资源投入。 |
| 回答持续抽象 | 启动前提挑战；仍说不清则标记待验证并继续。 |
