# 需求分析阶段质量门

阶段完成并推进到“需求拆解”前，必须逐条通过以下校验。

---

## 文件存在性

- [ ] `docs/requirement-analysis/req-001.md` 存在
- [ ] `docs/requirement-analysis/epic-001.md` 存在
- [ ] 至少一个 `docs/requirement-analysis/feature-*.md` 存在
- [ ] 需求分析阶段正式产物未分散写入 `docs/strategic/` 或 `docs/requirement/`

---

## Frontmatter 完整性

每份文档必须包含：

- [ ] `id`
- [ ] `type`
- [ ] `projectId`
- [ ] `title`
- [ ] `status`
- [ ] `refs`（Epic、Feature 至少一条引用关系；需求卡片可为空数组）

---

## 标题与写作质量

- [ ] 每份文档标题都是自然业务表达，能说明用户、场景或业务结果
- [ ] 每份文档至少包含背景、现状、目标、边界、指标、风险/待验证事项中的关键内容
- [ ] 事实、推断、假设和待验证事项已区分，不把未经确认的判断写成事实
- [ ] 正式卡片不是一两句话摘要；主要章节有展开说明或表格支撑

---

## 诊断报告质量

- [ ] 用户已确认诊断报告
- [ ] 诊断报告已回答四个核心判断：给谁做、为什么做、值不值得做、需要哪些产品/能力
- [ ] 四个核心判断均有证据或待验证缺口说明，未用功能清单替代判断
- [ ] 诊断报告包含问题本质还原
- [ ] 诊断报告包含需求转化记录
- [ ] 诊断报告包含关键假设和待验证事项
- [ ] 诊断报告包含项目类型判断（`new | iteration | refactor`）
- [ ] 如果存在多个问题，诊断报告已说明问题关系、产品闭环、范围边界和版本组织
- [ ] 诊断报告包含需求成熟度评分（0-10 分）
- [ ] 评分依据清晰：Q1-Q7 回答质量、数据支撑、用户画像、边界异常、非功能基线、验收标准和优先级风险
- [ ] 低于 5 分时，`tracking-log.md` 已记录补充调研计划，且未进入正式文档产出
- [ ] 替代方案不少于 2 个，且包含成本、时间、风险、ROI、适用条件

---

## 需求卡片质量

- [ ] 需求卡片能追溯到四个核心判断，且没有只停留在功能清单
- [ ] 一句话结论能讲清“谁在什么场景下要达成什么结果”
- [ ] 业务背景与触发原因清楚，能说明需求来源和为什么现在要做
- [ ] 问题本质描述清晰，不超过 3 句话
- [ ] 当前流程或替代方案已还原到关键步骤，不只是写“人工处理/系统分散”
- [ ] 业务影响与损失包含影响对象、频次/规模、证据或待验证来源
- [ ] 多个问题同时成立时，已说明范围内问题、问题关系、共同闭环依据和暂缓范围
- [ ] 目标用户画像具体，包含核心用户、典型场景、痛点强度、切换阈值
- [ ] 当前替代方案及痛点已说明
- [ ] 需求转化记录已说明原始表述、识别问题、转化后需求、转化依据
- [ ] 关键假设已列出（至少 1 条）
- [ ] 成功指标可量化（1-3 个）
- [ ] 需求评估结果三维度齐全：价值、紧急度、可行性
- [ ] 每个评估维度都有理由
- [ ] 待验证事项已列出或明确无待验证项

---

## Epic 质量

- [ ] 需求背景引用了需求卡片
- [ ] 产品名称已填写
- [ ] 产品定位一句话说清
- [ ] 产品目标可量化
- [ ] 端到端业务闭环已说明，包含关键环节、责任角色、动作、产出和改善结果
- [ ] 战略价值明确
- [ ] “为什么现在做”讲清楚业务时机、管理压力、用户诉求或风险窗口
- [ ] 目标用户/角色已列出
- [ ] 核心场景至少 1 个
- [ ] 多个问题同属一个产品时，已说明它们的共同用户、流程、数据对象或管理目标
- [ ] 边界清晰，明确范围内和范围外
- [ ] 建设思路包含分阶段路径
- [ ] 成功指标可量化

---

## Feature 质量

- [ ] 需求背景引用了需求卡片
- [ ] 能力名称已填写
- [ ] 能力描述清晰
- [ ] 能力目标可量化
- [ ] 用户任务用“作为...我希望...以便...”写清角色、场景、动作和价值
- [ ] 用户角色引用了 Epic
- [ ] 业务场景已列出（1-3 个）
- [ ] 使用前后对比已说明当前方式、目标方式和改善结果
- [ ] 业务价值说明充分
- [ ] 业务流程已描述到关键步骤、触发条件、操作角色、系统行为和输出结果
- [ ] 输入输出数据已说明来源/去向、用途和质量要求
- [ ] 业务规则已列出
- [ ] 异常分支已列出，包含系统处理和用户补救动作
- [ ] 技术可行性已评估
- [ ] 资源投入已估算（人天、人周或依赖资源）
- [ ] 优先级已标注（P0/P1/P2）
- [ ] 依赖关系已说明
- [ ] 验收标准可验证

---

## 数据校验

- [ ] 所有写入 `facts.json` 的数据已标注来源类型：`manual-input`、`file-extract`、`web-reference` 或 `derived-and-confirmed`
- [ ] 文件提取的数据经过格式校验和范围校验
- [ ] 数字口径与文档表述一致
- [ ] 校验异常数据已隔离，且在诊断报告“数据校验异常”章节列出
- [ ] 二手数据已标注来源和时间，经用户确认后才采纳
- [ ] 无来源标注的数据未被采纳

---

## 对抗性自审

五维度评分必须均不低于 7 分：

- [ ] 完整性：关键字段、引用关系、诊断链条齐全
- [ ] 一致性：需求卡片、Epic、Feature 之间无矛盾
- [ ] 清晰度：用户、场景、问题、目标可理解
- [ ] 范围：产品闭环、范围边界和版本组织明确，无平台化范围膨胀
- [ ] 可行性：技术、数据、资源、流程前提已识别
- [ ] 无未解决的 `validation-failed` 项
- [ ] 迭代轮次不超过 3 轮

---

## 用户确认

- [ ] 用户已确认诊断报告（含需求转化记录和成熟度评分）
- [ ] 用户已确认数据校验异常项（逐条核实或修正）
- [ ] 用户已确认方案、方案组合或分阶段路径
- [ ] 用户已确认需求卡片、Epic 和 Feature 内容

---

## 记忆更新

- [ ] `facts.json` 已记录已确认事实，且每条事实有来源
- [ ] `decision-log.md` 已记录方案选择、组合方式、理由和被否定或暂缓范围
- [ ] `tracking-log.md` 已记录未验证假设、风险、未决问题

- [ ] `progress.json` 已更新文档列表和阶段内进度
