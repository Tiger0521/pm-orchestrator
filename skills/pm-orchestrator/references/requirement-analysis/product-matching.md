## 产品匹配与复用引导（需求分析 intake）

本流程服务于触发"需求分析"时的项目 intake：在正式进入需求卡片、Epic、Feature 草稿前，帮助用户判断本次需求是否能复用已有产品。完成项目类型确认后，后续需求拆解承接已确认 Feature，详细设计承接已确认 Story/Feature，各自沿阶段 reference 推进。

本文件是产品匹配任务的唯一方法 reference。产品匹配任务（handoff `task` 限定为"只做产品匹配与项目类型建议"）只读本文件 + `product-library-spec.md`，**不读 `instruction.md`**（`instruction.md` 是需求草稿任务的 reference，含 686 行字段 JSON/落盘/质量门禁等草稿专用内容，产品匹配不需要）。

### 输入与读取顺序

1. 先读取用户自己的业务目标、初始描述和 `docs/background/` 中的背景材料摘要。用户未提供背景材料时，明确记录为"无前置背景材料"，不要虚构领域事实。
2. 再按 `product-library-spec.md` §8 渐进式披露流程读取产品库：先用 `_product.md` 生成候选导览（Step 2.5），用户选定候选后只读该候选需求卡片（Step 3a），早停判断通过才读 Epic（Step 3b），不匹配即停。只有进入 `refactor` 判断时才补读 User Story。产品匹配时同步核对从 `productArchitectureDesignPath` 读取的总体架构设计：候选产品是否符合产品定位、能力边界、数据口径、复用方式和演进方向。
3. 读取产品库不是为了让用户判断陌生产品，而是为了把已有产品翻译成用户能理解的业务语言，再围绕用户自己的需求事实做核对。

### 多候选产品导览

如果产品库中存在多个 low 及以上候选产品，不要立即深挖某一个产品。先给用户一个轻量候选导览，让用户选择先了解哪个候选。

**导览数据源**：只使用 `_product.md` 的 `summary`/`businessDomain`/`keywords` 三字段（领域初筛 Step 2 已读取，此处复用，不额外读卡片、不重复读 `_product.md`）。导览表只有 3 列：

| 候选产品 | 它解决的问题 | 建议优先级 |
| --- | --- | --- |
| `{product.id}` + `{product.name}` | `_product.md` 的 `summary` | 按 D6 领域初筛分值给出 high/medium/low |

**导览降级声明**：导览阶段的"建议优先级"仅基于 D6 领域初筛，不计算 D1-D5 深度匹配分。深度匹配留到用户选定候选后的 Step 3 执行。不得在导览阶段尝试算 D1/D2/D3，否则又需读需求卡片，破坏渐进式披露。

导览表不包含"可能重叠的业务闭环""可能扩展的方向"两列--这两列需读 Epic/Feature 正文，留到用户选定候选后的"单个已有产品解读"阶段再展示。

导览后只问一个选择题：

```text
我先找到了几个可能相关的已有产品。我会先帮你拆解它们分别覆盖什么，再根据你的业务事实判断复用空间。
你想先了解哪一个候选？

A. {候选 1}：{一句话说明为什么可能相关}
B. {候选 2}：{一句话说明为什么可能相关}
C. {候选 3}：{一句话说明为什么可能相关}
D. 补充描述：我自己填写
E. 强制跳过：暂不看候选产品，记录为待验证并继续
```

### 单个已有产品解读

用户选择候选，或只有一个 high/medium 候选时，按"需求卡片 -> Epic -> Feature"递进展示已有产品。展示已有产品后，进入业务事实核对：用户回答自己熟悉的业务时点、对象、数据、规则、流程和验收口径，系统据此推导覆盖点、差异点和项目类型建议。

**承接导览删列**：此阶段才展示导览阶段省略的"可能重叠的业务闭环""可能扩展的方向"两列内容，数据来自已读 Epic/Feature 正文。

1. **需求卡片层**：说明已有产品要解决的业务问题、现状痛点、问题本质、评估结果。
2. **Epic 层**：说明已有产品的产品定位、目标用户、核心场景、范围边界和建设思路。
3. **Feature 层**：说明已有产品已经有哪些能力，哪些能力可能直接覆盖用户需求，哪些可能通过扩展覆盖。

每一层都先批量展示多个字段的状态，再问一个最高价值问题。字段状态表建议使用：

| 字段 | 用户当前需求 | 已有产品对应内容 | 复用/扩展假设 | 状态 |
| --- | --- | --- | --- | --- |
| 问题本质 | `{用户表述}` | `{已有产品字段}` | `{相同/相近/需扩展}` | 已确认/待验证/缺失 |
| 目标用户 | `{用户表述}` | `{已有产品字段}` | `{同大类角色，可新增子角色}` | 已确认/待验证/缺失 |
| 核心场景 | `{用户表述}` | `{已有产品字段}` | `{同业务闭环或上下游}` | 已确认/待验证/缺失 |

### 业务事实核对写法

每轮问题围绕用户一开始需求中的关键字段展开。系统先提出复用假设，再问一个用户能直接确认的业务事实；用户回答事实，系统在回合后更新复用判断。

问题由四段组成：

1. **用户需求复述**：用用户自己的业务语言复述本次目标。
2. **已有产品解读**：说明已有产品覆盖的业务问题、场景或能力。
3. **系统复用假设**：说明系统当前推测可以复用什么、差异可能落在哪类扩展上。
4. **业务事实问题**：只问一个事实字段，例如业务时点、资源对象、目标角色、数据来源、校验动作、规则口径、流程边界或验收结果。

选项写成业务事实短句，让用户选择自己真实的业务情况。推荐句式：

```text
你的需求是「{用户需求复述}」。
我注意到已有产品已经覆盖「{已有产品业务语言解释}」。
我的初步判断是：可以先尝试复用「{已有 Epic/Feature/能力}」，差异可能落在「{扩展方向}」。

本轮我先核对一个业务事实：{围绕需求卡片/Epic/Feature 字段的问题}

A. {业务事实选项 1，例如发生在某业务时点/面向某角色/处理某类资源对象}
B. {业务事实选项 2，例如发生在另一个业务时点/接入另一类数据源/采用另一种规则口径}
C. {业务事实选项 3，例如覆盖更完整流程/更高频场景/更大组织范围}
D. {当前事实还不确定，需要先按待验证记录}
E. 补充描述：我自己填写
F. 强制跳过：这个问题暂时不回答，记录为待验证并继续
```

系统根据用户选择更新"已有产品已覆盖 / 本次新增或变化 / 仍待确认"的判断，再决定下一轮核对哪个字段。

### 扩展优先判断

做产品匹配时先发散地寻找"如何在已有产品上扩展"。字段差异优先进入扩展解释，常见扩展方向包括：

| 差异类型 | 优先解释 |
| --- | --- |
| 角色不同 | 先判断是否同一大领域角色；若是，通常是新增目标角色或权限视角 |
| 资源对象不同 | 先判断是否同资源模型的子类、扩展属性或新对象类型 |
| 规则不同 | 先判断是否新增校验规则、阈值、例外规则或规则版本 |
| 数据来源不同 | 先判断是否新增数据源、接口、采集口径或字段映射 |
| 流程不同 | 先判断是否新增流程分支、审批节点、处理状态或编排方式 |
| 场景不同 | 先判断是否已有核心场景的上下游、前后置或子场景 |
| 入口不同 | 先判断是否新增工作台、菜单、任务入口或角色视图 |
| 统计分析不同 | 先判断是否新增指标、报表、看板或预警口径 |
| 组织范围不同 | 先判断是否新增组织层级、区域范围、租户或权限边界 |

只有当共同问题本质、业务闭环、目标用户链路和核心数据对象都无法与已有产品建立合理关系时，才倾向 `new`。

### 收敛为项目类型

经过需求卡片、Epic、Feature 层的核对后，再结合从 `productArchitectureDesignPath` 读取的总体架构设计汇总立项建议：

- 倾向 `iteration`：已有产品的问题本质和业务闭环成立，本次差异主要落在角色、对象、规则、数据源、流程、场景、入口、统计或权限扩展。
- 倾向 `refactor`：业务闭环和产物定义应沿用已有产品，但用户明确指出现有方案架构、性能、稳定性、体验或规则实现存在系统性问题，需要推倒重来或非功能性改造。
- 倾向 `new`：本次需求的业务目标、用户链路、核心对象或价值主张与已有产品不同，继续挂接会造成产品定位混乱或重复约束。

输出给主调度器的结论必须包含：候选产品、匹配等级、已读产品资产范围、覆盖点、差异点、推荐项目类型、总体架构设计影响判断，以及仍需记录到 `decision-log.md` 的用户选择理由。

### 交互与输出格式

遵守主调度器 handoff 中的 `interactionContract`。

每次向用户提出下一问前，必须先给出 2-5 行"当前理解回执"：已确认了什么、还缺什么、当前追问属于整体问题地图的哪个位置。

每轮只能提出一个需要用户回答的问题或选择题。选择题选项必须使用大写英文字母顺序编号（`A.`、`B.`、`C.`、`D.`...），每个选择题必须包含两个固定兜底选项：`补充描述：我自己填写` 和 `强制跳过：这个问题暂时不回答，记录为待验证并继续`。

产品匹配任务完成时返回 `status=draft-ready`，在回执中给出 `projectType` 建议和匹配依据，等主调度器确认 `projectType` 后再次委派时才进入需求草稿。
