# 需求分析写作范式

## 用途

本文件为需求分析阶段所有落盘字段定义严格的语言范式。agent 在输出需求卡片、Epic、Feature 的每个字段时，必须对照本文件选择对应范式并严格遵循。

范式管的是"怎么写"——文字的结构、句式和丰富度。不管"写什么"——字段内容和追问流程仍由 `instruction.md` 和 `question-bank.md` 决定。

---

## 总则：六条通用规律

不论用哪种范式，所有字段的文字都必须遵守以下六条。任一条不满足，视为字段质量不达标。

### 1. 总结先行

每个字段的第一句话是一句总结性判断，概括这个字段要说的核心意思。不要上来就直接罗列条目。

- **差**：`1. 人工核对不可持续：... 2. 前清后乱顽疾：...`
- **好**：`当前网络资源的入网交维管理存在长期、具体的困难：` 然后分条

### 2. 加粗关键词领条

分条列点时，每条以 `**加粗关键词**` 开头，后接冒号或句号和详细解释。关键词是该条的"标签"，让读者扫一眼就能抓住要点。关键词应是有信息量的判断或意象，不是泛词。

- **差**：`1. 人工核对不可持续：一条光缆系统...`
- **好**：`- **人工核对是灾难**：一条光缆系统包含大量资源对象，靠人工逐条核对既不现实也无精力...`
- **差关键词**：`问题一`、`要点1`、`人工核对`（太泛）
- **好关键词**：`人工核对是灾难`、`"前清后乱"顽疾`、`流程与生产"两张皮"`（有判断、有意象）

### 3. 具名细节

解释里使用具体的部门名、工具名、人名、频次、数字，不用抽象词。能量化就量化，不能量化就举一个具名例子。

- **差**：`交维流程以线下方式为主，效率低`
- **好**：`交维流程以钉钉、邮件、纸质签字为主，资源数据由建设方手工录入资源管理系统`
- **差**：`定期组织清查`
- **好**：`每一两年都会组织大规模资源数据清查`
- **差**：`一线用户有迫切需求`
- **好**：`以广东省分公司管线专业人员为代表的一线同事，迫切希望把光缆入网交维流程从线下管控到线上`

### 4. 结论跟"为什么"

每个结论后面跟理由或因果链。用"不是…而是…"、"因为…所以…"、"…的原因是…"等句式把推理过程写出来。读者看到结论后脑中冒出的第一个问题应该是"为什么"，而"为什么"的答案就在下一句。

- **差**：`选择与现有系统完全隔离`
- **好**：`选择与现有系统**完全隔离**。现有系统由外包厂家分前后端建设，存在严重的组织与技术双重债务：厂家持续裁员、能力下滑、排斥新技术……在 capex 投资收紧的背景下，继续依赖厂家建设难以协调推进。`

### 5. 诚实标注

区分事实、推断和待验证内容。用 blockquote（`>`）标注边界声明、关键假设、责任划分。无法确认的内容写"待验证"，不假装确定。

- **好**：`> **诚实边界**：系统研发启动于 2026 年 4 月，"通用骨架对所有场景复用"是已被光缆入网验证的设计主张，而非对全部现网场景的承诺。`
- **好**：`> 责任边界说明：本系统作为下游功能支撑平台，对业务质量指标提供度量能力，不自行设定考核口径。`

### 6. 视觉结构匹配内容

内容有多条并列要点时用列表；有结构化对比时用表格；有流程或流转时用 mermaid 或 text 流程图。不要把所有信息压成一段流水句。选择哪种视觉结构由内容决定：

| 内容形态 | 视觉结构 |
| --- | --- |
| 多条并列要点（痛点、价值、目标） | 加粗关键词领条的列表 |
| 角色对比、规则对比、评估对比 | 表格（含关联列，不能只有名称+描述） |
| 流程、流转、状态转换 | mermaid flowchart 或 text 流程图 |
| 分层、分档、分优先级 | 小标题 + 每层表格或分条 |
| 深度论证（从事实到结论的推理） | 两段递进段落 |

## 六种语言范式

### 范式 A · 总结开头 + 分条列点

最基础、最高频的范式。适用于"有多个并列要点需要展开"的字段。

**结构**：

```
一句话总结性判断（概括这些要点的共性）：

- **关键词1**：详细解释，含具名细节和因果
- **关键词2**：详细解释，含具名细节和因果
- **关键词3**：详细解释，含具名细节和因果
```

**规则**：
1. 第一句是总结判断，不是"以下有几个要点"这类空话
2. 至少 2 条，每条独立成段
3. 每条以 `**加粗关键词**` 开头，冒号后跟解释
4. 解释里至少包含一个具名细节（人/工具/频次/数字）

**适用字段**：痛点、产品目标、产品价值、业务价值、范围外、业务场景

---

### 范式 B · blockquote 核心论断 + 分要点展开

适用于"需要先给出一个核心论断，再拆解其内涵"的字段。

**结构**：

```
> **一句话核心论断（加粗，放在 blockquote 中）**

这句话有 N 个要点：

1. **要点关键词**。解释段落，含因果和具名细节。
2. **要点关键词**。解释段落，含因果和具名细节。
3. **要点关键词**。解释段落，含因果和具名细节。
```

**规则**：
1. 核心论断用 blockquote + 加粗，是整个字段的"题眼"
2. 论断后用"这句话有 N 个要点"过渡到分条
3. 要点用编号列表（`1.` `2.` `3.`），每条以加粗关键词开头，后接句号和解释段落
4. 解释段落要比范式 A 更充分，每个要点是一个完整的论证段落

**适用字段**：产品定位

---

### 范式 C · 开头定位 + 表格

适用于"有多个同构对象需要对比"的字段。

**结构**：

```
一句话说明分类维度和逻辑：

| 列1 | 列2 | 列3（关联列） |
| --- | --- | --- |
| 值 | 值 | 值 |
```

**规则**：
1. 表格上方必须有一句话说明"按什么维度分类"或"覆盖什么范围"
2. 表格至少 3 列，其中最后一列应为"关联列"——对应理由、痛点、异常处理等，不能只有名称+描述两列
3. 单元格内容不能只有标签，要有一句话解释
4. 表格下方可跟 blockquote 补充说明或边界声明

**适用字段**：需求评估结果、用户角色、建设思路、业务规则、资源投入

**变体 C-表头字段**：需求基本信息也是表格形式（key-value），每个 value 应遵循"定性 + 具名细节"的写法，不能只写标签。

---

### 范式 D · 开头定位 + 流程图 + 关键特征分条

适用于"描述流程、现状或场景"的字段。

**结构**：

```
一句话定位该流程/现状/场景在整个业务中的位置：

```mermaid
flowchart ...
```

关键特征：

- **特征关键词1**：解释
- **特征关键词2**：解释
```

**规则**：
1. 开头一句话定位——这个流程在整体业务中处于什么位置，上承什么、下接什么
2. 流程图用 mermaid `flowchart` 或 text 箭头图，节点用业务语言命名
3. 流程图后用"关键特征"或"关键步骤"分条补充图中无法表达的重点
4. 如果流程有异常分支，图中用条件标注（`-->|校验不通过|`）

**适用字段**：现状描述、核心场景、业务流程

---

### 范式 E · 分层结构 + 每层表格/分条

适用于"需要分层、分档展示"的字段。

**结构**：

```
一句话说明分层逻辑：

**第一层 · 层名**

（表格或分条）

**第二层 · 层名**

（表格或分条）

**第三层 · 层名**

（表格或分条）
```

**规则**：
1. 开头一句话说明"按什么维度分层"
2. 每层用 `**层名**` 小标题领起
3. 每层内用表格或分条，内容结构一致
4. 层与层之间有递进或优先级关系，不是平级罗列

**适用字段**：建设思路（P0/P1/P2 分档）

---

### 范式 F · 段落论证

适用于"需要从事实推导出结论"的字段。

**结构**：

```
第一段：摆事实（现状、证据、具名细节，不含结论）

第二段：引出结论（"这说明…"、"因此…"、"这意味着…"）

> （可选）blockquote 标注更深层的意义或边界
```

**规则**：
1. 第一段只摆事实和证据，不下结论
2. 第二段用过渡词（"这说明"、"因此"、"这意味着"）引出结论
3. 结论要回答"本质是什么"或"因此该怎么做"
4. 可选 blockquote 标注更深层的意义、边界或假设

**适用字段**：问题本质还原、技术可行性

---

## 字段范式速查表

### 简单字段

以下字段内容简短，不需要套用复杂范式，但仍需遵守总则中的"具名细节"和"结论跟为什么"：

| 字段 | 所属文档 | 写法要求 |
| --- | --- | --- |
| 需求背景 | 需求卡片 / Epic / Feature | 引用上游文档，一句话概括核心问题；自动填入，无需追问 |
| 产品名称 | Epic | 业务语言命名，非技术代号；一句话说清它是什么 |
| 能力名称 | Feature | 业务语言命名，能对应真实用户任务；非菜单名或模块名 |
| 用户角色（Feature） | Feature | 引用 Epic 角色，说明本 Feature 由哪个角色使用、谁查看、谁审批 |
| 优先级 | Feature | `P0/P1/P2` + 排序依据（"因为…，不先做会阻塞…"） |

以下字段必须严格套用对应范式：

| 字段 | 所属文档 | 范式 |

## 各字段范式详解与好差对比

### 需求卡片

---

#### 需求基本信息

**范式**：C-表头字段（表格，每个 value 用"定性 + 具名细节"）

**规则**：表格中每个字段的值不能只写一个标签，要写"一句话定性 + 具名细节"。

**差写法**：

```
| 需求来源 | 双向坐实 |
| 提出人/角色 | 集团副总、一线人员 |
| 触发时间/时机 | 近期事项 |
| 影响范围 | 四个专业 |
| 当前状态 | 尚无系统支持 |
```

**好写法**：

```
| 需求来源 | 双向坐实——自上而下（集团副总战略要求、中心年度网络治理工作方案）与自下而上（广东省分公司管线人员迫切诉求）两股驱动力方向一致，同时存在 |
| 提出人/角色 | 集团副总经理（自上而下推动全流程贯通）、中心网络治理工作组（制定年度方案）、广东省分公司管线专业人员（自下而上迫切诉求） |
| 触发时间/时机 | 2026 年中心网络治理工作方案落地周期，集团要求四专业系统贯通与数据贯通 |
| 影响范围 | 光缆、传输、动环、无线四个专业，涉及建设方/外包厂家、维护方/运维团队、一线管线专业人员 |
| 当前状态 | 现有资源全生命周期系统由外包厂家建设，存在组织与技术双重债务，无源头数据质量管控能力 |
```

---

#### 现状描述

**范式**：D（开头定位 + 流程图 + 关键特征）

**规则**：
1. 开头一句话定位当前业务在整个链条中的位置
2. 用 mermaid 或 text 流程图画出当前流程的关键步骤
3. 用"关键特征"或"关键问题"分条补充图中无法表达的重点

**差写法**：

```
当前流程：智能规建推送交维数据 -> 系统生成入网交维工单 -> 按专业/场景/网络层级路由到对应审批流程 -> 多级审批 -> 配置化数据校验（查询+比对+阈值判定）-> 校验通过则归档并推送资源中心，校验不通过则退回补充/修正后重新校验
```

**好写法**：

```
当前网络资源的入网交维管理散落在多个系统和线下工具中，缺乏统一的源头管控：

```text
智能规建系统（建设竣工）
    ↓ 钉钉/邮件通知交维
建设方手工录入资源数据到资源管理系统
    ↓ 纸质签字审批
维护方人工逐条核对资源数据
    ↓ 核对通过则交维完成
资源数据进入网络资源中心（事后可能不准）
```

关键问题：

- **交维靠线下、靠人情**：交维流程以钉钉、邮件、纸质签字为主，资源数据由建设方手工录入，"录得全不全、对不对"一线想知道但缺手段。
- **人工核对不可持续**：一条光缆系统包含大量资源对象，靠人工逐条核对既不现实也无精力，质量把关在人情关系下容易流于形式。
- **前清后乱**：集团与各省每一两年组织大规模资源数据清查，清查完很快又乱。现有的数据治理平台只能事后清理，无法在数据产生的源头管控。
```

---

#### 痛点

**范式**：A（总结开头 + 分条列点）

**规则**：
1. 第一句总结痛点的性质（如"存在长期、具体的困难"）
2. 每条以 `**加粗关键词**` 开头，冒号后跟具体行为 + 频次/规模 + 损失后果
3. 至少 2 条，每条独立成段

**差写法**：

```
1. 人工核对不可持续：一条光缆系统包含大量资源对象，靠人工逐条核对既不现实也无精力，质量把关在人情关系下容易流于形式。2. 前清后乱顽疾：集团与各省每1-2年组织大规模资源数据清查，清查完很快又乱。现有数据治理平台只能事后清理，无法在数据产生的源头管控
```

> 差在哪：没有总结开头；编号后没有加粗关键词；两条压成一段；缺少损失后果。

**好写法**：

```
当前网络资源的入网交维管理存在长期、具体的困难：

- **资源是"哑资源"**：光缆等资源缺乏与网管采集数据的有效比对手段，资源数据是否准确、完整无从校验。导致政企业务开通核查时可能误判"资源不具备"，重复发起建设投资。
- **交维靠线下、靠人情**：交维流程以钉钉、邮件、纸质签字为主，资源数据由建设方手工录入资源管理系统，"录得全不全、对不对、和设计图纸是否一致"一线想知道但缺手段。
- **人工核对是灾难**：一条光缆系统包含大量资源对象，靠人工逐条核对既不现实也无精力，质量把关在人情关系下容易流于形式。
- **"前清后乱"顽疾**：集团与各省每一两年都会组织大规模资源数据清查，清查完很快又乱。现有的数据治理平台只能事后清理，无法在数据产生的源头管控。
- **流程与生产"两张皮"**：一线的生产作业已通过钉钉、邮件或本地自建审批完成，但这些流程与资源数据更新并不联动。若系统要求一线干完活后再到系统里重新走一遍流程，对一线就是与实际生产脱节的额外负担，结果多沦为事后补单。
```

---

#### 问题本质还原

**范式**：F（段落论证）

**规则**：
1. 第一段摆事实：当前的具体现象和证据，不含结论
2. 第二段引出结论：用"这说明"、"因此"过渡，回答"本质是什么"
3. 可选 blockquote 标注更深层的意义或边界

**差写法**：

```
源头失控：数据质量管控不在数据产生的第一时间介入，而是事后补救。资源数据在录入和交维环节缺乏有效的质量门禁，导致脏数据流入系统后只能靠周期性清查治理，治理完又很快被新数据污染，形成前清后乱的恶性循环
```

> 差在哪：只有结论没有事实铺垫；一段压缩句，没有推理过程；没有"这说明/因此"的过渡。

**好写法**：

```
当前资源数据的录入和交维环节缺乏有效的质量门禁。交维流程以钉钉、邮件、纸质签字为主，资源数据由建设方手工录入，"录得全不全、对不对"一线想知道但缺手段。集团与各省每一两年组织大规模清查，但清查完很快又乱——现有数据治理平台只能在数据已经进入资源中心之后做事后清理，无法在数据产生的源头管控。

这说明问题的本质是**源头失控**：数据质量管控不在数据产生的第一时间介入，而是事后补救。脏数据流入系统后只能靠周期性清查治理，治理完又很快被新数据污染，形成"前清后乱"的恶性循环。要打破这个循环，必须在数据进入资源中心之前设立不可绕过的质量关卡。

> 更深一层：这不只是数据质量问题，而是管控方式问题。过去的思路是"先污染后治理"，用清查补漏；正确的方式是"源头把关"，让不合格数据在产生时就进不了系统。
```

#### 能力描述

**范式**：F（段落，从用户视角描述前后状态变化）

**规则**：
1. 第一段描述用户使用该能力前的状态和困难
2. 第二段描述使用后能完成什么、状态变成什么
3. 从用户视角写，不从系统功能视角写

**差写法**：

```
用户输入设备名称或别名后，系统匹配候选设备，展示端口数量、端口占用、连接关系、数据来源和更新时间。
```

> 差在哪：只有系统功能描述，没有用户前后状态变化；没有"之前怎样"的铺垫。

**好写法**：

```
一线资源管理员在故障排查时，只知道设备名称，无法快速查到端口数量、端口占用和设备间连接关系，需要跨多个系统和 Excel 台账人工拼接，平均耗时 30-60 分钟。

使用本能力后，用户输入设备名称或别名，系统在 1 分钟内返回候选设备的端口数量、端口占用、连接关系，并标注每条数据的来源系统和更新时间。用户无需跨系统查询和手工拼接，直接在统一入口获取可信结果。
```

#### 能力目标

**范式**：A（总结 + 分条，含量化指标）

**规则**：
1. 第一句总结能力目标的方向
2. 每条以加粗关键词开头，含"从X到Y"的量化变化
3. 至少 1 条可量化的指标

**差写法**：

```
让核心设备查询从 30 分钟降至 1 分钟以内，且每条结果都有来源标注。
```

> 差在哪：只有一句话，没有分条；没有总结开头。

**好写法**：

```
本能力要实现以下可验证的目标：

- **查询效率**：核心设备查询从 30-60 分钟降至 1 分钟以内。
- **结果可信度**：每条查询结果标注来源系统和更新时间，来源标注覆盖率 100%。
- **减少二次修正**：查询结果被人工二次修正比例低于 10%。
```

#### 业务价值

**范式**：A（总结开头 + 分条列点）

**规则**：
1. 第一句总结价值方向
2. 每条以加粗关键词开头，含对用户和业务的具体价值
3. 至少 1 条说明"不做会损失什么"

**差写法**：

```
减少跨系统查询和 Excel 拼接，降低错误判断风险。
```

> 差在哪：只有半句话；没有分条；没有"不做会损失什么"。

**好写法**：

```
本能力对用户和业务的价值：

- **减少查询耗时**：一线资源管理员不再需要跨多个系统和 Excel 台账人工拼接，单次查询节省 30-50 分钟。高峰期每天查询 10 次以上，累计节省显著。
- **降低错误判断风险**：查询结果带来源标注，用户可判断数据可信度，避免基于过时或不一致数据做错误判断。
- **为后续治理沉淀数据链路**：查询日志和异常数据登记为后续资源数据治理提供可信的数据基础。不做则数据口径不一致问题持续存在，故障定位时效无法改善。

#### 业务场景

**范式**：A（总结 + 分条，每条含触发条件和用户目标）

**规则**：
1. 第一句总结场景覆盖范围
2. 每条以加粗场景名开头，含"什么时候触发 + 用户要完成什么"
3. 至少 1 个场景

**差写法**：

```
1. 故障排查时快速确认设备端口占用。2. 割接前核实设备间连接关系。3. 发现数据异常时提交资源待办。
```

> 差在哪：没有总结开头；编号后没有加粗关键词；缺少"用户从什么状态到什么状态"的描述。

**好写法**：

```
本能力在以下业务场景中触发：

- **故障排查**：运维工程师收到故障告警后，需要快速确认故障设备的端口占用和上联关系，以判断故障影响范围和定位根因。从"跨系统查 30 分钟"到"1 分钟内查到"。
- **割接前核实**：割接方案制定前，工程师需要核实涉及设备间的连接关系，确认割接影响范围。从"靠 Excel 台账核对"到"系统直接展示"。
- **数据异常登记**：查询过程中发现数据异常时，用户直接提交资源待办，从"发现问题但无人跟进"到"异常进入待办闭环"。
```

#### 业务流程

**范式**：D（开头定位 + 流程图 + 关键步骤）

**规则**：
1. 开头一句话定位该流程的触发条件和最终结果
2. 用 mermaid 或 text 流程图画出关键步骤
3. 用分条补充异常分支和兜底策略

**差写法**：

```
输入设备名称 -> 匹配候选设备 -> 展示端口与连接关系 -> 标注来源 -> 异常提交待办
```

> 差在哪：只有一行箭头流程；没有开头定位；没有异常分支；没有加粗关键词。

**好写法**：

```
用户从输入设备名称到获取查询结果，中间经过匹配、展示和异常处理：

```text
用户输入设备名称/别名
    → 系统匹配候选设备（精确匹配优先，别名和模糊匹配作为候选）
    → 展示端口数量、端口占用、连接关系
    → 每条结果标注来源系统和更新时间
    → 用户判断数据是否可信
        → 数据正常：查询完成
        → 数据异常：提交资源待办，进入待验证清单
```

关键步骤说明：

- **名称匹配**：精确匹配优先。如果有多个候选设备，展示候选列表让用户选择。如果无匹配结果，提示用户检查名称或尝试别名。
- **来源标注**：每条端口和连接关系必须显示来源系统和更新时间。如果某字段无来源数据，显示"数据缺失"而非留空。
- **异常隔离**：数据冲突时不得覆盖主数据，只进入待验证清单。用户可标注冲突原因，流转给数据责任人处理。
```

**范式**：C（表格，含异常处理列）

**规则**：
1. 表格至少 3 列：规则项、规则说明、异常/边界处理
2. 异常处理列不能为空，要写"如果规则被违反，系统怎么处理"

**差写法**：

```
名称匹配：精确匹配优先，别名和模糊匹配作为候选。来源标注：每条数据必须显示来源。异常隔离：不覆盖主数据。

> 差在哪：没有表格；没有异常处理列；压缩成一段。

**好写法**：

```
| 规则项 | 规则说明 | 异常/边界处理 |
| --- | --- | --- |
| 名称匹配 | 精确匹配优先，别名和模糊匹配作为候选 | 无匹配结果时提示用户检查名称或尝试别名；多个候选时展示列表让用户选择 |

## JSON 编码注意事项

字段 JSON 中的值是字符串，多行 markdown 内容（列表、表格、mermaid 代码块）需要在 JSON 中用 `\n` 表示换行。`render-doc.sh` 会将 `\n` 转换为实际换行符后输出到 Markdown 文件。

**示例**：痛点字段的 JSON 值：

```json
{
  "pain_points": "当前网络资源的入网交维管理存在长期、具体的困难：\n\n- **资源是\"哑资源\"**：光缆等资源缺乏与网管采集数据的有效比对手段，资源数据是否准确、完整无从校验。\n- **交维靠线下、靠人情**：交维流程以钉钉、邮件、纸质签字为主，资源数据由建设方手工录入。"
}
```

**规则**：
1. JSON 字符串中的换行用 `\n`，段落间距用 `\n\n`
2. JSON 字符串中的双引号用 `\"` 转义
3. mermaid 代码块中的反引号在 JSON 中不需要特殊转义，但整个代码块内容需在同一 JSON 字符串值内
4. 表格中的 `|` 字符在 JSON 中不需要转义
5. 草稿预览（对话中展示）与字段 JSON 和最终渲染的 Markdown 三者必须同结构、同字段、同正文内容
