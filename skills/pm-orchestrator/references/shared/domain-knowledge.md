# 联通网络资源管理领域知识

本文件为需求分析阶段的领域知识底座。涉及网络资源管理业务时按需加载，用于让追问和诊断切中业务要害。

## 业务定位

工具服务于中国联通网络资源管理部门，聚焦网络资源全生命周期管理。该部门面向后域网络设备管理，管辖入网网络设备，覆盖设备监控、资源调拨、跨部门协调，以及网络连接、数据承载等相关业务。

设计目标不是通用需求分析，而是适配运营商网络资源管理场景：理解设备、光缆、基站、机房等资源的管理差异，理解从规划到入网再到运维的流程约束，理解一线用户通常不懂技术、只知道设备名称的使用习惯。

## 核心资源类型

| 资源类型 | 说明 | 管理特点 |
| --- | --- | --- |
| 设备 | 入网网络设备 | 需全生命周期管理，任何变更都应可追溯 |
| 光缆（管线资源） | 通信光缆 | 典型无源资源，只需铺设，不需额外供电 |
| 基站（无线资源） | 由天线、CU、AU 等多个设备组成的功能单元 | 可测算，各类无线设备均需管理 |
| 空间资源 | 联通机房 | 需记录面积、位置等信息，常存在手工录入 |
| 行政区域数据 | 与行政区域维护相关的数据 | 作为资源定位、归属和统计的基础维度 |

## 资源管理理念

资产投入资金后转变为资源，资源需要运维运行：运营整个网络、保障网络安全、确保资源处于可用状态。需求分析时要优先追问“资源是否完整、准确、可用”，而不是只追问界面或功能。

## 全生命周期管理

典型流程：

```text
规划 → 项目立项 → 施工 → 采购 → 正式接入网络 → 运维
```

运维阶段包括设备更新、零部件更换、设备搬迁、软件更新、业务割接、扩容等。每种流程都会影响设备数据，例如扩容后内存从 8G 变为 16G。此类数据不能随意修改，必须有流程依据和变更记录。

## 数据治理机制

入库数据通常经过：

```text
合规性检查 → 比对流程 → 资源待办 → 全生命周期管理
```

- 比对流程以自动比对为主，网络配置等场景可人工操作。
- 比对发现异常时进入资源待办，由人工核实。
- 采集解析的数据可能有误，系统应允许人工修正并保留来源。
- 需求分析时要追问数据来源、口径、更新频率、异常处理和人工兜底。

## 数据共享方式

| 方式 | 特点 | 追问重点 |
| --- | --- | --- |
| 接口服务 | 可随时查询调用 | 接口稳定性、权限、实时性、调用方 |
| 只读数据库 | 直接查询分析 | 表结构、口径一致性、查询权限 |
| 文件共享 | 定期上传供下载分析 | 更新周期、格式校验、版本管理 |

## 工作目标三层级

| 层级 | 目标 | 需求诊断重点 |
| --- | --- | --- |
| 第一层 | 保证数据完整 | 所有应纳管资源是否都在系统中留存 |
| 第二层 | 保证数据准确 | 资源属性、状态、关系是否可信 |
| 第三层 | 场景化应用支持 | 用户能否在具体场景中快速使用数据 |

## 核心痛点

### 数据更新繁琐

用户不愿手动逐行填写大量字段。要追问哪些字段必须人工确认、哪些可从文件或系统同步、哪些字段可以后置补齐。

### 数据查询困难

一线用户不懂技术，往往只知道设备名称，想快速查询设备端口数量、端口占用情况、设备间连接关系等。理想状态是通过自然语言或极简查询完成。

### 跨系统协同成本高

资源数据、流程审批、网络配置、项目建设和运维记录可能分布在多个系统。需求分析时要追问系统边界、数据归属、接口可用性和人工交接点。

### 数据可信度不足

采集、解析、人工录入、历史迁移都可能造成脏数据。需求分析时要追问校验规则、异常隔离、人工复核和事实来源标注。

## 典型追问方向

- 这个需求涉及哪类资源：设备、光缆、基站、机房、行政区域，还是组合关系？
- 资源当前处于生命周期哪个环节：规划、建设、入网、运维、退网？
- 数据来源是什么：人工录入、采集解析、接口、只读库、文件？
- 哪些字段必须准确，哪些字段可以暂时待验证？
- 数据异常后谁处理？进入什么待办？是否影响网络安全或业务开通？
- 一线用户只知道设备名称时，系统如何定位到端口、链路、位置或关系？
